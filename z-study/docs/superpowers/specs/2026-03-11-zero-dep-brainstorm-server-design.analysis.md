## 二、目的

将脑暴伴侣服务器从一个"有 700+ 依赖"的内嵌实现，重构为一个**单文件、零依赖、可审计、可教学**的 Node.js 应用。

### 三大动机

1. **供应链风险**：内嵌 node_modules 不会被自动打安全补丁
2. **审计盲区**：714 个第三方文件未经审计就入仓
3. **修改难以识别**：vendored code 的修改看起来像普通 commit，绕过依赖审查流程

虽然实际风险低（仅 localhost 开发服务器），但消除风险是简单可行的。

---

## 三、核心技术决策

### 1. 单一文件双角色

`server.js` 同时支持：
- `node server.js` → 启动服务器
- `require('./server.js')` → 导出协议函数供测试

**实现**：`if (require.main === module)` 判断入口模式。

**价值**：测试不需要启动整个 HTTP 服务器——可纯单元测试协议层。

### 2. WebSocket 协议的"最小可工作子集"

| 特性 | 实现 | 跳过原因 |
|---|---|---|
| 文本帧（0x01） | ✅ | 脑暴用 JSON 文本 |
| 关闭帧（0x08） | ✅ | 基础协议要求 |
| Ping/Pong（0x09/0x0A） | ✅ | 基础协议要求 |
| 二进制帧（0x02） | ❌ | 脑暴不用 |
| 分片消息 | ❌ | 小负载用不到 |
| 扩展（permessage-deflate） | ❌ | 压缩不必要 |
| 子协议 | ❌ | 不声明则不激活 |

**价值**：**只实现需要的，其余不实现**——避免过度工程。

### 3. 三种长度编码必须支持

WebSocket 协议规定三种 payload 长度编码：
- `< 126` 字节：1 字节长度
- `126-65535` 字节：126 标记 + 16 位长度
- `> 65535` 字节：127 标记 + 64 位长度

**为何不能简化**：客户端可能发送任意大小消息（虽然脑暴场景小，但要"按 RFC 实现"）。

### 4. 文件监听替代 chokidar

**之前**：`chokidar`（跨平台、API 复杂、依赖大）
**之后**：`fs.watch`（Node 内置、API 简单）

**风险缓解**：100ms 防抖处理 macOS/Linux 的重复事件。

### 5. 双 stdout / file 输出

服务器启动时既向 stdout 输出 JSON，又写入 `.server-info` 文件。

**为何需要文件输出**：agent 在后台执行时 stdout 可能被隐藏，文件作为冗余通道。

### 6. 路由极简（仅 3 条）

| 路由 | 作用 |
|---|---|
| `GET /` | 渲染屏幕（最新 .html） |
| `GET /files/*` | 静态文件服务 |
| 其他 | 404 |

**没有中间件**——`express` 不是必需的。

### 7. 错误处理的"日志+继续"哲学

```
- 格式错误 JSON → 记录到 stderr，继续
- 未处理 opcode → 关闭连接（仅关闭该连接）
- fs.watch 错误 → 记录到 stderr，继续
```

**不抛出、不崩溃**——让服务器"在错误中继续运行"。

### 8. 不做优雅停机

服务器不做 SIGTERM 处理。**`stop-server.sh` 脚本负责进程管理**。

**理由**：边界职责清晰——Node 进程只负责运行，shell 脚本只负责启停。

## 六、缺点 / 风险

| 风险点 | 影响 | 缓解 |
|---|---|---|
| **扩展性受限** | 不支持 binary frame/permessage-deflate | 当前脑暴场景纯 JSON 文本 |
| **fs.watch 平台差异** | macOS/Linux/Windows 事件行为略不同 | 防抖 + error handler |
| **维护成本转移** | 协议 bug 修复需自己移植 | 代码量 < 300 行，bug 易于排查 |
| **测试需要 `ws` 客户端** | 集成测试仍需 ws 库 | 测试目录的 package.json 保留依赖（不入仓） |
| **非优雅停机** | SIGTERM 后连接被立即中断 | 当前脑暴场景可接受 |
| **无内建中间件** | CORS、auth 需手动实现 | localhost-only，无需 |

## 七、关键洞察

### 洞察 1："实现规范"与"实现规范子集"的智慧

完全实现 RFC 6455 要数千行代码。但脑暴只用一小部分。**"实现需要的，跳过不需要的"**是务实工程的核心。

### 洞察 2：单文件双角色的优雅

```js
if (require.main === module) {
  startServer();
}
```

3 行代码就让一个文件既是 CLI 入口又是模块。**不强制"一个文件一个角色"**是 Node 的优雅。

### 洞察 3：双通道输出（stdout + 文件）的鲁棒性

服务器启动信息既向 stdout 输出，又写入文件——**任何一种通道失败，另一种仍可用**。

这是"defense in depth"在启动信息中的体现。

### 洞察 4：依赖洁癖的元价值

vendored node_modules 的真正问题不是技术问题，而是**心理问题**——"它就在那里，看起来像我们自己的代码"。**删除依赖是组织治理**。

### 洞察 5：明确边界 = 简化责任

不实现优雅停机看起来是缺陷，实则是**边界清晰**——Node 进程做运行，shell 脚本做管理。**不让一个组件承担所有职责**。

### 洞察 6：测试集成 vs 测试纯净的折中

集成测试用 `ws` 库——**测试可使用便利工具，但产品代码不引入**。这是"test-only dependency"模式。

## 十、给读者的启示

### 启示 1：协议学习的最佳途径是亲手实现
**RFC 规范 → 第三方库 → 亲手实现**——300 行代码胜过 30 页规范。

### 启示 2：单文件不等于简陋
300 行的单文件可包含完整协议实现 + HTTP 服务器 + 文件监听 + 静态服务。**代码组织靠内部分层，不靠文件数**。

### 启示 3："实现子集"是务实工程
完全实现 RFC 是过度工程。**只实现需要的，跳过不需要的**是工程成熟的标志。

### 启示 4：依赖是方便也是锁链
每个 `npm install` 都是对上游的信任投票。**审视哪些依赖真的"必要"**——很多时候"必要"是"习惯"。

### 启示 5：测试可使用便利工具
集成测试用 `ws` 客户端库没问题——**测试可引入便利，但产品代码要节制**。

### 启示 6：边界清晰 > 全能组件
不做优雅停机不是缺陷，是**职责清晰**。Node 进程只管运行，shell 脚本管生命周期。

### 启示 7：双通道输出的鲁棒性
启动信息既 stdout 又文件——**任何通道失败不影响 agent 启动**。这是简单的"defense in depth"。

## 十一、后续工作（推测）

1. **二进制帧支持**：若脑暴需要传图/音频，需扩展 BINARY opcode 处理
2. **permessage-deflate 扩展**：当负载增大时启用压缩
3. **TLS 支持**：未来若需要非 localhost 部署，需 `https` 模块
4. **CORS/Auth 中间件**：当扩展到多用户场景
5. **`fs.watch` 替换为 `fs.watchFile`**：处理嵌套目录监听需求
6. **优雅停机**：当 `stop-server.sh` 之外有更精细的进程管理需求时
7. **TypeScript 化**：将 `server.js` 转为 `.ts`，加类型检查
8. **协议层独立 npm 包**：未来若脑暴服务器用于其他项目，可发布为独立包
