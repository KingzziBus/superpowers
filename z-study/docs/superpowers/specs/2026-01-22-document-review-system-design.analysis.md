## 二、目的

在 spec 和 plan 文档的关键节点插入"质量门控"，让 superpowers 工作流从"写完即用"升级为"写完即审"。

### 三个核心痛点

1. **缺乏反馈循环**：作者写完 spec 后直接进入 plan，无客观校验
2. **TODO 残留**：作者倾向于"先写完再补 TODO"——这些 TODO 经常被遗忘
3. **spec-plan 脱节**：plan 阶段重新阅读 spec 时才发现理解偏差

---

## 三、核心技术决策

### 1. 复用子智能体机制

不引入新工具，而是通过 Task 工具派发 `general-purpose` 子智能体。

**价值**：
- 零基础设施投入
- 子智能体隔离主对话上下文
- 审查 prompt 是 markdown 模板，可版本控制和迭代

### 2. 审查器的"顾问性"原则

- 审查器**标记**问题但**不阻塞**
- 智能体若认为审查器反馈错误，可解释并继续
- 同一问题 3 轮后仍分歧 → 上抛人类

**为什么不是阻塞式**：
- 审查器可能误报
- 作者应有最终决定权
- 避免被"完美主义"困住

### 3. 三层上抛机制

| 触发条件 | 动作 |
|---|---|
| 审查循环 > 5 轮 | 上抛人类 |
| 同一分歧 > 3 轮 | 上抛人类 |
| 格式错误 > 2 次 | 上抛人类 |

**价值**：避免无限循环，明确"何时停止"的判断标准。

### 4. Markdown 任务语法标准化

- `### Task N: Name`
- `- [ ] **Step 1:** ...`

**目的**：
- 进度可视化
- 工具可解析
- 整个项目格式契约

### 5. Chunk 边界（plan 审查的关键概念）

```
## Chunk N: <name>
```

**为什么按 chunk 审查**：
- 单个 plan 可能 5000+ 行，无法一次性审查
- chunk 是逻辑自包含的（如 "Foundation"、"Core Features"）
- 每个 chunk < 1000 行

## 七、缺点 / 风险

| 风险点 | 影响 | 缓解 |
|---|---|---|
| **依赖子智能体可用性** | 非 Claude Code 平台可能无 Task 工具 | 计划注明 "if subagents available" |
| **审查器质量** | 通用子智能体未必熟悉 spec 评审 | 详细"What to Check" + "CRITICAL" 重点 |
| **审查开销** | 每个 spec 走 1-2 轮 = 多 1-2 次子智能体 | 顾问性原则 + 上抛阈值 |
| **advisory 易被滥用** | 作者可能无理由否决 | 必须给出解释 |
| **事后拦截** | 写完才审，无法预防 | 设计权衡 |

## 八、关键洞察

### 洞察 1：审查是"软件工程防御性编程"扩展

Linter 在代码层防止 bug；Reviewer 在文档层防止缺陷。**任何产出都值得被另一双眼睛看一眼**。

### 洞察 2：审查模板是知识资产

`spec-document-reviewer-prompt.md` 不只是 prompt，更是**项目对"什么是好 spec"的具象定义**。任何团队成员读这两个文件就能学会评审标准。

### 洞察 3：元层级终止条件

"5 轮上限"把"何时停止"从工程权衡提升为流程规则。**给任何循环结构都加最大迭代次数**是工程文化成熟的标志。

### 洞察 4：Spec 和 Plan 的"双向验证"

- Spec 审查器检查 spec 本身的质量
- Plan 审查器检查 plan 与 spec 的对齐

**双向验证让"spec → plan"链条上的每一步都被审视**。

### 洞察 5：Markdown 任务语法的元价值

复选框语法 `### Task N:` + `- [ ] **Step:**` 看似简单，实则建立了**项目文档格式的契约**——杠杆效应巨大。

## 十一、给读者的启示

### 启示 1：质量门控可以"无侵入"
**把现有资源（Task 工具 + 通用子智能体）组合到工作流关键节点**——就能搭建出高效门控。

### 启示 2：模板化是知识沉淀的最佳载体
评审标准写在 `prompt.md` 里比写在 wiki 中更可执行。**让流程标准以可执行文件形式存在**。

### 启示 3：设定元层级的终止条件
"5 轮上限"看似不起眼，实则是显性规则。**给任何循环结构都加最大迭代次数**。

### 启示 4：审查与创作是互补的
本设计不是"用审查取代创作"，而是"创作 + 审查 + 修复"闭环。**让 agent 既是作者也是审稿人**。

### 启示 5：双向验证让链条更鲁棒
- Spec 审查器：spec 内部质量
- Plan 审查器：spec-plan 对齐

**每个转换节点都设置检查点 = 鲁棒的工程流程**。

### 启示 6：advisory 而非 blocking
审查器应是**顾问**而非**裁判**——保留作者决定权，避免"完美主义陷阱"。

## 十二、后续工作（推测）

3. **审查报告归档**：将每次审查的 issues/recommendations 写入 git commit message 或独立报告
4. **审查者模型选择**：未来可考虑专用 reviewer-only 模型
