# 来自用户反馈的技能改进

**日期：** 2025-11-28
**状态：** 草稿
**来源：** 两个 Claude 实例在真实开发场景中使用 superpowers

---

## 执行摘要

两个 Claude 实例从实际开发会话中提供了详细反馈。他们的反馈揭示了当前技能中**系统性的缺口**，这些缺口导致可预防的 bug 流出，即使遵循了相关技能。

**关键洞察：** 这些是问题报告，不仅仅是解决方案建议。问题是真实的；解决方案需要仔细评估。

**关键主题：**
1. **验证缺口** - 我们验证操作成功，但不验证它们是否达到预期结果
2. **流程卫生** - 后台进程累积并在子代理之间相互干扰
3. **上下文优化** - 子代理收到太多无关信息
4. **缺乏自我反思** - 在交接前没有提示来审视自己的工作
5. **Mock 安全性** - Mock 可能在没有检测的情况下偏离接口
6. **技能激活** - 技能存在但未被读取/使用

---

## 已识别的问题

### 问题 1：配置变更验证缺口

**发生了什么：**
- 子代理测试"OpenAI 集成"
- 设置了 `OPENAI_API_KEY` 环境变量
- 得到 200 状态响应
- 报告"OpenAI 集成工作"
- **但是** 响应包含 `"model": "claude-sonnet-4-20250514"` - 实际上使用的是 Anthropic

**根本原因：**
`verification-before-completion` 检查操作成功，但不检查结果是否反映预期的配置变更。

**影响：** 高 - 对集成测试的错误自信，bug 流入生产

**失败模式示例：**
- 切换 LLM 提供商 → 验证状态 200 但不检查模型名称
- 启用功能开关 → 验证无错误但不检查功能是否激活
- 更改环境 → 验证部署成功但不检查环境变量

---

### 问题 2：后台进程累积

**发生了什么：**
- 会话期间分派了多个子代理
- 每个子代理启动后台服务器进程
- 进程累积（4+ 个服务器在运行）
- 旧进程仍绑定到端口
- 后续 E2E 测试命中错误配置的旧服务器
- 混乱/不正确的测试结果

**根本原因：**
子代理是无状态的 - 不知道之前子代理的进程。没有清理协议。

**影响：** 中-高 - 测试命中错误的服务器，错误通过/失败，调试混乱

---

### 问题 3：子代理提示中的上下文膨胀

**发生了什么：**
- 标准方法：给子代理完整计划文件来读
- 实验：只给任务 + 模式 + 文件 + 验证命令
- 结果：更快、更聚焦、更常一次完成

**根本原因：**
子代理在无关的计划章节上浪费 token 和注意力。

**影响：** 中 - 执行更慢，更多失败尝试

**有效的方式：**
```
你正在为 packnplay 的测试套件添加单个 E2E 测试。

**你的任务：** 在 `pkg/runner/e2e_test.go` 中添加 `TestE2E_FeaturePrivilegedMode`

**要测试的内容：** 一个在元数据中请求 `"privileged": true` 的本地 devcontainer 特性，
应导致容器使用 `--privileged` 标志运行。

**完全遵循 TestE2E_FeatureOptionValidation 的模式**（在文件末尾）

**写完后运行：** `go test -v ./pkg/runner -run TestE2E_FeaturePrivilegedMode -timeout 5m`
```

### 问题 4：交接前缺乏自我反思

**发生了什么：**
- 添加了自我反思提示："用全新的眼光审视你的工作 - 有什么可以做得更好？"
- 任务 5 的实施者识别出失败的测试是由于实现 bug 而不是测试 bug
- 追溯到第 99 行：`strings.Join(metadata.Entrypoint, " ")` 创建了无效的 Docker 语法
- 没有自我反思的话，只会报告"测试失败"而没有根本原因

**根本原因：**
实施者在报告完成前不会自然地退后一步审视自己的工作。

**影响：** 中 - 传递给审查者的 bug 是实施者本可以自己捕获的

### 问题 5：Mock-接口 偏离

**发生了什么：**
```typescript
// Interface defines close()
interface PlatformAdapter {
  close(): Promise<void>;
}

// Code (BUGGY) calls cleanup()
await adapter.cleanup();

// Mock (MATCHES BUG) defines cleanup()
vi.mock('web-adapter', () => ({
  WebAdapter: vi.fn().mockImplementation(() => ({
    cleanup: vi.fn().mockResolvedValue(undefined),  // 错误！
  })),
}));
```
- 测试通过
- 运行时崩溃："adapter.cleanup is not a function"

**根本原因：**
Mock 来自有 bug 的代码调用，而不是来自接口定义。TypeScript 无法捕获具有错误方法名的内联 mock。

**影响：** 高 - 测试给出错误自信，运行时崩溃

**为什么 testing-anti-patterns 没有阻止这一点：**
该技能涵盖了测试 mock 行为和不理解的情况下使用 mock，但没有覆盖"从接口而非实现派生 mock"的具体模式。

### 问题 6：代码审查者的文件访问

**发生了什么：**
- 分派代码审查子代理
- 找不到测试文件："该文件似乎在仓库中不存在"
- 文件实际存在
- 审查者不知道要先显式读取它

**根本原因：**
审查者提示不包含显式文件读取指令。

**影响：** 低-中 - 审查失败或不完整

---

### 问题 7：修复工作流延迟

**发生了什么：**
- 实施者在自我反思期间识别出 bug
- 实施者知道修复方法
- 当前工作流：报告 → 我分派修复者 → 修复者修复 → 我验证
- 多余的往返增加了延迟却没有增加价值

**根本原因：**
当实施者已经诊断出问题时，实施者和修复者角色的僵化分离。

**影响：** 低 - 延迟，但没有正确性问题

---

### 问题 8：技能未被读取

**发生了什么：**
- `testing-anti-patterns` 技能存在
- 人类和子代理在写测试前都没有读它
- 本可以防止一些问题（尽管不是全部 - 见问题 5）

**根本原因：**
没有强制要求子代理读取相关技能。没有提示包含技能读取。

**影响：** 中 - 技能投入被浪费

## 拟议的改进

### 1. verification-before-completion：添加配置变更验证

**添加新章节：**

```markdown
## 验证配置变更

测试配置、提供商、功能开关或环境的变更时：

**不要只验证操作成功。验证输出反映预期的变更。**

### 常见失败模式

操作成功，因为*某一些*有效配置存在，但它不是你想要测试的配置。

### 示例

| 变更 | 不充分 | 需要 |
|------|--------|------|
| 切换 LLM 提供商 | 状态 200 | 响应包含预期的模型名称 |
| 启用功能开关 | 无错误 | 功能行为实际激活 |
| 更改环境 | 部署成功 | 日志/变量引用新环境 |
| 设置凭据 | 认证成功 | 已认证用户/上下文正确 |

### 门控函数

```
在声明配置变更工作之前：

1. 识别：变更后应该有什么不同？
2. 定位：在哪里可以观察到这种差异？
   - 响应字段（模型名称、用户 ID）
   - 日志行（环境、提供商）
   - 行为（功能激活/未激活）
3. 运行：显示可观察差异的命令
4. 验证：输出包含预期差异
5. 然后：声明配置变更工作

红旗：
  - "请求成功"但未检查内容
  - 检查状态码但不检查响应体
  - 验证无错误但无正面确认
```

**为什么有效：**
强制验证意图，而不仅仅是操作成功。

### 2. subagent-driven-development：为 E2E 测试添加流程卫生

**添加新章节：**

```markdown
## E2E 测试的流程卫生

当分派启动服务（服务器、数据库、消息队列）的子代理时：

### 问题

子代理是无状态的 - 它们不知道之前子代理启动的进程。后台进程持续存在，可能会干扰后续测试。

### 解决方案

**在分派 E2E 测试子代理之前，在提示中包含清理：**

```
在启动任何服务之前：
1. 杀死现有进程：pkill -f "<service-pattern>" 2>/dev/null || true
2. 等待清理：sleep 1
3. 验证端口空闲：lsof -i :<port> && echo "错误：端口仍在使用" || echo "端口空闲"

测试完成后：
1. 杀死你启动的进程
2. 验证清理：pgrep -f "<service-pattern>" || echo "清理成功"
```

### 示例

```
任务：运行 API 服务器的 E2E 测试

提示包括：
"在启动服务器之前：
- 杀死任何现有服务器：pkill -f 'node.*server.js' 2>/dev/null || true
- 验证端口 3001 空闲：lsof -i :3001 && exit 1 || echo '端口可用'

测试后：
- 杀死你启动的服务器
- 验证：pgrep -f 'node.*server.js' || echo '清理已验证'"
```

### 为什么这很重要

- 旧进程使用错误的配置服务请求
- 端口冲突导致静默失败
- 进程累积减慢系统
- 测试结果混乱（命中错误的服务器）
```

**权衡分析：**
- 给提示增加样板代码
- 但防止非常混乱的调试
- 对 E2E 测试子代理值得

### 3. subagent-driven-development：添加精简上下文选项

**修改步骤 2：使用子代理执行任务**

**之前：**
```
从 [plan-file] 仔细阅读该任务。
```

**之后：**
```
## 上下文方法

**完整计划（默认）：**
当任务复杂或有依赖关系时使用：
```
仔细阅读 [plan-file] 中的任务 N。
```

**精简上下文（用于独立任务）：**
当任务是独立的且基于模式时使用：
```
你正在实现：[1-2 句任务描述]

要修改的文件：[精确路径]
要遵循的模式：[对现有函数/测试的引用]
要实现的内容：[具体要求]
验证：[要运行的确切命令]

[不要包含完整计划文件]
```

**在以下情况使用精简上下文：**
- 任务遵循现有模式（添加类似测试、实现类似功能）
- 任务是自包含的（不需要其他任务的上下文）
- 模式引用已足够（例如，"遵循 TestE2E_FeatureOptionValidation"）

**在以下情况使用完整计划：**
- 任务依赖于其他任务
- 需要了解整体架构
- 需要上下文的复杂逻辑
```

**示例：**
```
精简上下文提示：

"你正在为 devcontainer 特性中的特权模式添加测试。

文件：pkg/runner/e2e_test.go
模式：遵循 TestE2E_FeatureOptionValidation（在文件末尾）
测试：元数据中带 `"privileged": true` 的特性导致 `--privileged` 标志
验证：go test -v ./pkg/runner -run TestE2E_FeaturePrivilegedMode -timeout 5m

报告：实现、测试结果、任何问题。"
```

**为什么有效：**
减少 token 使用，提高聚焦度，在适当时更快完成。

### 4. subagent-driven-development：添加自我反思步骤

**修改步骤 2：使用子代理执行任务**

**添加到提示模板：**

```

退后一步，用全新的眼光审查你的工作。

问问自己：
- 这是否真正解决了指定的任务？

**为什么有效：**
在交接前捕获实施者自己能发现的 bug。文档化案例：通过自我反思识别入口点 bug。

**权衡：**
每个任务增加约 30 秒，但在审查前捕获问题。

### 6. testing-anti-patterns：添加 Mock-接口 偏离反模式

**添加新反模式 6：**

```markdown
## 反模式 6：从实现派生的 Mocks

**违规：**
```typescript
// 代码（有 bug）调用 cleanup()
await adapter.cleanup();

// Mock（与 bug 匹配）有 cleanup()
const mock = {
  cleanup: vi.fn().mockResolvedValue(undefined)
};

// 接口（正确）定义 close()
interface PlatformAdapter {
  close(): Promise<void>;
}
```

**为什么这是错误的：**
- Mock 将 bug 编码到测试中
- TypeScript 无法捕获具有错误方法名的内联 mock
- 测试通过，因为代码和 mock 都是错误的
- 使用真实对象时运行时崩溃

**修复方法：**
```typescript
// ✅ 良好：从接口派生 mock

// 步骤 1：打开接口定义（PlatformAdapter）
// 步骤 2：列出其中定义的方法（close、initialize 等）
// 步骤 3：Mock 正好是这些方法

const mock = {
  initialize: vi.fn().mockResolvedValue(undefined),
  close: vi.fn().mockResolvedValue(undefined),  // 来自接口！
};

// 现在测试失败，因为代码调用 cleanup()，该方法不存在
// 该失败在运行时之前揭示了 bug
```

### 门控函数

```
在编写任何 mock 之前：

  1. 停止 - 暂时不要查看被测试的代码
  2. 查找：依赖项的接口/类型定义
  3. 阅读：接口文件
  4. 列出：接口中定义的方法
  5. Mock：仅 Mock 正好是这些方法
  6. 不要：查看你的代码调用了什么

  如果你的测试失败，因为代码调用了 mock 中没有的内容：
    ✅ 良好 - 测试发现了你代码中的 bug
    修复代码以调用正确的接口方法
    而不是 mock

  红旗：
    - "我将 mock 代码调用的内容"
    - 从实现复制方法名
    - 未阅读接口就编写 mock
    - "测试失败，所以我会向 mock 添加此方法"
```

**检测：**

当你看到运行时错误"X 不是函数"且测试通过时：
1. 检查 X 是否被 mock
2. 比较 mock 方法和接口方法
3. 查找方法名不匹配
```

**为什么有效：**
直接解决反馈中的失败模式。

**权衡：**

### 8. subagent-driven-development：允许实施者修复自我识别的问题

**修改步骤 2：**

**当前：**
```
子代理报告工作摘要。
```

**拟议：**
```
子代理执行自我反思，然后：

如果自我反思识别出可修复的问题：
  1. 修复问题
  2. 重新运行验证
  3. 报告："初始实现 + 自我反思修复"

否则：
  报告："实现完成"

包含在报告中：
- 自我反思发现
- 是否应用了修复
- 最终验证结果
```

**为什么有效：**
当实施者已经知道修复时减少延迟。文档化案例：本来可以为入口点 bug 节省一次往返。

**权衡：**
提示略复杂，但端到端更快。

   - 我们如何处理进程应该持久化的情况（开发服务器）？

4. **技能读取强制：**
   - 我们应该要求所有子代理读取相关技能吗？
   - 我们如何防止提示变得过长？
   - 过度文档化和失去聚焦度的风险？

---

## 成功指标

我们如何知道这些改进有效？

1. **配置验证：**
   - 零次"测试通过但使用了错误配置"的情况
   - Jesse 不说"那实际上并没有测试你认为的内容"

2. **流程卫生：**
   - 零次"测试命中错误的服务器"
   - E2E 测试运行期间无端口冲突错误

3. **Mock-接口 偏离：**
   - 零次"测试通过但运行时因缺少方法而崩溃"
   - Mock 和接口之间无方法名不匹配

4. **自我反思：**
   - 可衡量：实施者报告是否包含自我反思发现？
   - 定性：进入代码审查的 bug 是否更少？

5. **技能读取：**
   - 子代理报告引用技能门控函数
   - 代码审查中反模式违规更少

## 风险与缓解措施

### 风险：提示膨胀
**问题：** 添加所有这些要求使提示不堪重负
**缓解：**
- 分阶段实施（不要一次添加所有）
- 使一些添加有条件（仅 E2E 测试的卫生）
- 考虑不同任务类型的模板

### 风险：分析瘫痪
**问题：** 过多的反思/验证减慢执行
**缓解：**
- 保持门控函数快速（秒级，而非分钟）
- 最初将精简上下文设为可选
- 监控任务完成时间

### 风险：错误的安全感
**问题：** 遵循清单不能保证正确性
**缓解：**
- 强调门控函数是最小值，不是最大值
- 在技能中保留"使用判断力"语言
- 记录技能捕获常见失败，而不是所有失败

### 风险：技能分歧
**问题：** 不同技能给出相互冲突的建议
**缓解：**
- 审查所有技能的变更是否一致
- 记录技能如何交互（集成章节）
- 在部署前用真实场景测试

## 建议

**立即进入阶段 1：**
- verification-before-completion：配置变更验证
- testing-anti-patterns：Mock-接口 偏离
- requesting-code-review：显式文件读取

**在最终确定前用 Jesse 测试阶段 2：**
- 获取自我反思影响的反馈
- 验证流程卫生方法
- 确认技能读取要求是否值得开销

**暂挂阶段 3，等待验证：**
- 精简上下文需要真实世界测试
- 实施者修复工作流变更需要仔细评估

这些变更解决了用户记录的真实问题，同时最大限度地降低使技能变差的风险。
