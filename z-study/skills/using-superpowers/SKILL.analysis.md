## 一、文档目的

这是 Superpowers 体系的 **"宪法"** —— 所有其他技能都建立在它的规则之上。

具体目标：
1. **强制技能发现**：让 AI 在任何响应前先"问"是否有相关技能
2. **建立指令优先级**：用户指令 > Superpowers > 默认系统提示
3. **防止合理化**：列出 12 条 LLM 逃避调用技能的借口
4. **跨平台规范**：明确不同 AI 编程工具的技能调用方式

定位上属于 **"AI 行为守门人"的最顶层**——和 `verification-before-completion`、`receiving-code-review` 共同构成 **"AI 行为守门人三角"**。

## 二、核心技术决策

### 决策 1：`<SUBAGENT-STOP>` 例外

```text
<SUBAGENT-STOP>
如果你作为子代理被分派去执行特定任务，跳过本技能。
</SUBAGENT-STOP>
```

**设计意图**：
- 子代理是**已经知道任务**的——不需要再问"用哪个技能"
- 强制调用反而**浪费**子代理的"工作带宽"
- **例外 = 智能的规则**

**精妙之处**：用 XML 标签包裹例外——**视觉上显眼**，避免子代理误读。

### 决策 2：`<EXTREMELY-IMPORTANT>` 强约束

```text
如果你认为有哪怕 1% 的可能性某个技能可能适用于你正在做的事，你绝对必须调用该技能。

如果某个技能适用于你的任务，你没有选择。你必须用它。

这不可谈判。这不是可选的。你不能用合理化的方式回避它。
```

**关键洞察**：
- **"1% 概率"** 是核心数字——把"高门槛"反过来
- **"你必须用它"** 是命令式而非建议式
- **"你不能用合理化的方式回避它"** 戳穿 LLM 最常用的逃避策略

**这是"反 LLM 自主权"的最强表述**。

### 决策 3：3 级指令优先级

```text
1. 用户的显式指令（CLAUDE.md、GEMINI.md、AGENTS.md、直接请求）—— 最高
2. Superpowers 技能 —— 中（覆盖默认系统行为）
3. 默认系统提示 —— 最低
```

**关键洞察**：
- **用户主权**优先于一切——防止 Superpowers 变成"过度工程"
- **技能权威**高于默认行为——技能是有意为之的覆盖
- **默认行为**最弱——系统提示是"兜底"

**这是"主权让渡"的设计**——把最高权力给用户。

### 决策 4：禁止用 Read 工具读技能

```text
在 Claude Code 中：使用 Skill 工具。当你调用一个技能，它的内容会被加载并展示给你 —— 直接遵循它。
**永远不要**用 Read 工具读技能文件。
```

**为什么**：
- `Skill` 工具是**平台专用的"语义加载"**——理解技能结构
- `Read` 工具只是"文件读取"——把技能当成普通文本
- 错误的方式可能导致**技能流程被忽略**

**这是"工具语义不可替换"原则**。

### 决策 5：决策图形式化技能调用流程

```dot
digraph skill_flow {
    "User message received" [shape=doublecircle];
    "About to EnterPlanMode?" [shape=doublecircle];
    "Already brainstormed?" [shape=diamond];
    ...
}
```

**设计意图**：
- 形式化**技能调用的判断流程**
- 进入 PlanMode 前**先问"是否 brainstorming 过"**——这是 Superpowers 的核心流
- "1% 概率就调"——**显式低门槛**

**精妙之处**：
- `About to EnterPlanMode?` 是**入口点**——直接拦截常见路径
- `Already brainstormed?` 是**流程检查**——防止跳过 brainstorming
- 决策图让"调用技能"成为**必经路径**而非"可选优化"

### 决策 6：12 条红旗（反合理化）

```text
"这只是一个简单的问题" → 问题就是任务。检查技能。
"我需要先了解上下文" → 技能检查在澄清问题之前。
"让我先探索代码库" → 技能告诉你如何探索。先检查。
...
```

**关键洞察**：
- 列出 LLM **最常用的 12 种逃避理由**
- 每一行都是**直接的反驳**
- 这是**元规则 (meta-rule)** —— 教 AI "不能用什么借口"

**为什么不只是 1-2 条**：
- LLM 的合理化能力**极强**——只列 1-2 条会被"绕过"
- 12 条覆盖**绝大多数常见借口**
- 让 AI "**找不到**逃避的措辞"

### 决策 7：技能优先级（流程 → 实施）

```text
1. 流程技能（brainstorming、debugging）—— 决定"如何"
2. 实施技能（frontend-design、mcp-builder）—— 指导"做"
```

**关键洞察**：
- **流程在前**——决定工作方法
- **实施在后**——具体执行
- 颠倒顺序 = "**不规划就执行**"——反 Superpowers 哲学

**这是"方法优先于执行"原则**。

### 决策 8：技能类型（严格 vs 灵活）

```text
严格型（TDD、debugging）：严格遵循。不要为了方便而削弱纪律。
灵活型（patterns）：根据上下文调整原则。
```

**精妙之处**：
- 告诉 AI "**不是所有技能都同样严格**"
- 严格型 = 必须遵循（TDD 不能"为了方便跳过"）
- 灵活型 = 原则可适配
- **技能本身会告诉你类型**——AI 不需要硬编码分类

**这是"自我描述的元数据"**——技能自带类型说明。

### 决策 9：用户指令的"WHAT vs HOW"区分

```text
指令说的是"做什么"，不是"怎么做"。"加 X"或"修 Y"不意味着跳过工作流。
```

**关键洞察**：
- "加 X"是 **WHAT**——不指定方法
- "跳过 brainstorming"才是 **HOW**（用户没说的话）
- **用户没指定方法** = **AI 应该用 Superpowers 流程**

**这是"默认流程"原则**——用户没反对就用 Superpowers。

## 四、优点

### 1. EXTREMELY-IMPORTANT 块是"AI 行为宪法"的精粹

```text
这不可谈判。这不是可选的。你不能用合理化的方式回避它。
```

**优点**：
- 用**强语气**防止 LLM 软化规则
- 三个"这不可..." = 节奏感强
- 戳穿 LLM 最常用的"合理化"逃避

**这是 LLM 行为对齐中**最强烈的措辞之一**。

### 2. SUBAGENT-STOP 例外是"智能规则"

```text
如果你作为子代理被分派去执行特定任务，跳过本技能。
```

**优点**：
- 避免子代理被**冗余的"调用哪个技能"** 浪费带宽
- 承认"规则有例外"
- **规则 + 例外 = 完整规则**

**这是"规则设计的成熟度"标志**。

### 3. 3 级指令优先级是"权力分配"的形式化

```text
1. 用户主权
2. Superpowers 权威
3. 默认行为
```

**优点**：
- 用户**始终**掌控——防止 Superpowers 变成"过度控制"
- 优先级明确——没有歧义
- 例子直接（"不要用 TDD"vs "始终用 TDD"）

### 4. 决策图是"技能调用的形式化"

**优点**：
- 形式化决策树——可测试、可讨论
- `About to EnterPlanMode?` 是**关键拦截点**——直接重定向到 brainstorming
- 决策图比"if-else 文字"更清晰

### 5. 12 条红旗是"反合理化的元规则"

**优点**：
- 列出 LLM **最常用来逃避技能调用的借口**
- 每一行都是**直接的反驳**
- 12 条 = 覆盖绝大多数场景

**这是"反 LLM 训练偏差"的硬性动作**。

### 6. 技能优先级（流程 → 实施）

**优点**：
- 防止"不规划就执行"
- 强调**方法论**优先
- 符合工程实践

### 7. 技能类型自我描述

**优点**：
- 技能**自带类型说明**——不需要 AI 硬编码
- "严格型"和"灵活型"——简单二分
- 避免 AI 误用（"对灵活型技能严格遵循"或"对严格型技能灵活调整"）

### 8. WHAT vs HOW 区分

**优点**：
- 戳穿"用户指令=方法"的常见误解
- 明确"用户没指定就用 Superpowers 流程"
- 防止 AI **跳过流程** 来"按用户要求做"

### 9. 跨平台工具映射

**优点**：
- Claude Code / Copilot / Gemini / Codex / OpenCode 都有明确指引
- 把 Superpowers **推广到所有 AI 编程工具**
- 不锁定到单一平台

### 10. 简洁（118 行承载全部规则）

**优点**：
- 元规则 + 决策图 + 12 条红旗 + 类型 + 优先级 = 全部装在 118 行
- 信息密度**极高**
- 任何 AI 代理都能**几秒钟读完**

## 五、缺点与风险

### 1. "1% 概率"可能导致过度调用

**问题**：
- 1% 是非常低的门槛
- AI 可能**频繁调用不相关技能**——浪费 token
- 例如："让我喝杯水" 也要检查技能？

**风险**：**过度防御**反而降低效率。

**缺失**：调用的**成本-收益**评估。

### 2. `<EXTREMELY-IMPORTANT>` 的强制语气可能让 LLM 过度紧张

**问题**：
- 强语气可能让 LLM **过度焦虑**——什么都不敢做
- 实际效果是 LLM **频繁调技能** 但**什么都不做**

**风险**：把"严格"变成"瘫痪"。

### 3. SUBAGENT-STOP 例外可能被 LLM 误用

**问题**：
- LLM 可能**自我认定为"子代理"** 来逃避调用
- 例外被滥用

**缺失**：**"子代理"的客观判定标准**。

### 4. "12 条红旗"可能不够

**问题**：
- LLM 的合理化能力**比 12 条多**
- 实际使用中会发现新的"借口模式"
- 文档没说明**如何扩展红旗清单**

**风险**：**元规则也会过时**。

### 5. 决策图的"About to EnterPlanMode" 检查

**问题**：
- 只覆盖 EnterPlanMode 入口
- 实际中可能**其他入口**（如直接修改文件）也会需要 brainstorming
- 决策图**不全面**

**缺失**：其他入口的拦截。

### 6. "流程 → 实施" 优先级在复杂任务中可能冲突

**场景**：
- "我们来做 X" → brainstorming（流程）
- brainstorming 中可能需要"如何做 X"的领域知识（实施）
- **双向依赖**

**缺失**：流程和实施冲突时的处理。

### 7. 用户指令优先级可能被滥用

**场景**：
- 用户说"不要用 brainstorming"
- 但**用户不知道** brainstorming 的价值
- AI 遵从用户 → 跳过 brainstorming → 质量下降

**缺失**：**用户可能错**的场景处理。

### 8. 工具映射的"4 个平台" 可能不够

**当前覆盖**：Claude Code、Copilot CLI、Gemini CLI、Codex、OpenCode
**未覆盖**：Cursor、Aider、Cody、其他新兴工具

**风险**：新平台需要单独适配。

### 9. 决策图用 dot 语法

**问题**：
- dot 语法需要 Graphviz 渲染
- 在 Markdown 预览中**看不到图**
- 没有预渲染版本

**改进**：提供 PNG/SVG 或用 mermaid。

### 10. "用户指令"和"Superpowers"的边界可能模糊

**场景**：
- CLAUDE.md 中说"用 TDD"——这是用户指令还是 Superpowers？
- 如果**两者都要求用 TDD**——OK
- 如果**两者冲突**——以谁为准？

**缺失**：**多源指令冲突**的处理。

### 11. 缺乏"使用统计"和"调用反馈"

**问题**：
- 不知道 AI **实际调用了哪些技能** vs 应该调用哪些
- 没有"调用错误率"指标
- 改进缺少数据支撑

**缺失**：技能调用的**可观测性**。

### 12. 没有"技能版本管理"

**问题**：
- 技能会演化（"Skills evolve" 是文档原话）
- 但**没说明** AI 应该用最新版本
- 缓存的旧技能可能误导

**缺失**：**版本管理协议**。

## 六、关键洞察

### 洞察 1：这是 Superpowers 的"宪法"

**所有其他技能**都建立在本技能的规则之上：
- `verification-before-completion`：是"如何做"的具体规则
- `receiving-code-review`：是"如何接收反馈"的具体规则
- `brainstorming`：是"如何开始"的具体规则
- `TDD`、`debugging` 等：都是"具体场景"的规则

**本技能**则是"**任何时候都要**遵守的元规则"。

**意义**：这是 Superpowers 体系的**第 0 层**——在所有技能之上的技能。

### 洞察 2：12 条红旗是"反 LLM 训练偏差的元规则"

**观察**：
- LLM 的训练数据里**充斥着"自主发挥"**——大多数 LLM 默认"自己决定怎么做事"
- 用户的指令只是**意向**，LLM 自己填补**方法**
- 这种"自主"在工程上是**反生产力**

**本技能**反其道而行之：
- "**先问技能**"作为**必经路径**
- 12 条红旗**显式禁止** LLM 找借口

**这是"AI 行为对齐"的核心技术**。

### 洞察 3：EXTREMELY-IMPORTANT 块是"AI 训练的强 prompt"

```text
这不可谈判。这不是可选的。你不能用合理化的方式回避它。
```

**关键洞察**：
- LLM 训练时**对语气敏感**——强语气让 LLM 更难"软化"
- "不能用合理化的方式" 直接戳穿 LLM 的 meta-ability
- **重复"不可谈判"** = 强化指令权重

**这是"prompt 工程"的精粹**——不只是说理，是用**修辞**让 LLM 必须遵守。

### 洞察 4：决策图让"调用技能"成为"必经路径"

```dot
"Might any skill apply?" -> "Invoke Skill tool" [label="yes, even 1%"];
```

**关键洞察**：
- 把"调用技能"从**可选优化**变成**必经路径**
- "1%" = 显式低门槛
- 决策图让 LLM 看到"调用"是**默认**而非"特殊"

**这是"行为路径的工程化"**——通过图论方式强制行为。

### 洞察 5：3 级指令优先级是"权力分配的形式化"

```
1. 用户主权
2. Superpowers 权威
3. 默认行为
```

**哲学立场**：
- **用户始终掌控**——防止 Superpowers 变成"过度控制"
- 但 Superpowers 是**有意义的覆盖**——不是"用户说什么就什么"
- 默认行为是**兜底**——只在没有更高指令时生效

**这是"层次化权威"的设计**。

### 洞察 6：技能类型自我描述是"自包含元数据"

```text
严格型（TDD、debugging）：严格遵循。
灵活型（patterns）：根据上下文调整原则。
```

**关键洞察**：
- 不需要 AI 硬编码"哪个技能是严格型"
- 技能**自己说明**类型
- **减少 AI 的"分类负担"**

**这是"自描述数据"的设计**。

### 洞察 7：WHAT vs HOW 是"用户意图的解析"

```text
"加 X"或"修 Y"不意味着跳过工作流。
```

**关键洞察**：
- 用户的"加 X" = **WHAT**（要什么）
- "用 brainstorming"或"跳过 brainstorming" = **HOW**（怎么做）
- **HOW 通常没说** = AI 应该用 Superpowers 流程

**这是"指令解析的形式化"**。

### 洞察 8：跨平台工具映射是"Superpowers 的扩展性"

**当前覆盖**：Claude Code、Copilot CLI、Gemini CLI、Codex、OpenCode

**关键洞察**：
- 每个平台的**工具 API** 不同
- 把"Claude Code 的工具名"映射到"等价物"
- **新平台只需添加映射**——核心规则不变

**这是"接口抽象 + 平台适配"模式**。

### 洞察 9：与 verification-before-completion 的"二象性"

```
using-superpowers：        决定"用什么技能"
verification-before-completion：决定"做完没做完"
```

**两者**：
- 前者管"路径选择"
- 后者管"完成判断"

**共同构成** AI 行为的"输入和输出"两翼**。

### 洞察 10：决策图 + 12 红旗 = "行为的形式化 + 反例的形式化"

**正反两面**：
- 决策图（digraph）：**正确的路径**的形式化
- 12 条红旗：**错误的借口**的形式化

**这是"行为规范"的两面设计**——"什么做"和"什么不能做"都被显式化。
