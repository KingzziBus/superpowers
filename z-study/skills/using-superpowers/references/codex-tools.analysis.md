## 一、文档目的

解决 Superpowers **跨平台**战略的具体技术问题：**让所有以 Claude Code 工具名编写的技能，在 Codex 上也能跑**。

具体目标：
1. **建立工具名映射**：Claude Code 的 `Task`/`TodoWrite`/`Skill` 在 Codex 上叫什么
2. **启用多代理支持**：Codex 默认可能没开 multi_agent
3. **环境检测的复用**：worktree 相关技能的 git 原语检测
4. **Codex App 特殊场景**：沙箱限制下的降级方案

定位上属于 **"运行时翻译层"**——和 `using-superpowers` 母技能配套，是**技能可移植性的基础设施**。

## 二、核心技术决策

### 决策 1：技能用 Claude Code 工具名为"标准"

```text
技能使用 Claude Code 的工具名。
```

**设计意图**：
- 统一"源语言"——所有技能以 Claude Code 工具名为基准
- 平台适配 = 把"源语言"翻译为"目标平台"
- 避免每平台一份技能（爆炸式维护成本）

**这是"i18n"模式应用到工具**——以一个平台为标准。

### 决策 2：表格形式呈现映射

| 技能引用 | Codex 等价物 |
|----------|--------------|
| `Task` 工具 | `spawn_agent` |
| TodoWrite | `update_plan` |
| ... | ... |

**优点**：
- 一目了然
- 便于 AI 检索
- 可扩展（加新行）

### 决策 3：多代理支持是"显式 opt-in"

```toml
[features]
multi_agent = true
```

**为什么**：
- Codex 默认可能没开 multi_agent（节省资源）
- `subagent-driven-development` 这类技能**必须**有 multi_agent
- 显式 opt-in = 防止意外性能开销

**这是"性能 vs 能力"的权衡**——让用户选择。

### 决策 4：保留 legacy 兼容性说明

```text
旧版说明：rust-v0.115.0 之前的 Codex 构建把分派代理的等待暴露为 `wait`。
当前 Codex 对分派代理使用 `wait_agent`。
```

**为什么**：
- 很多用户**还在用旧版 Codex**
- 不写旧版说明 = 旧用户困惑
- 显式说明"新版/旧版" = 跨版本兼容

**这是"工程文档的成熟度"**——不假设用户都升级了。

### 决策 5：环境检测代码复用

```bash
GIT_DIR=$(cd "$(git rev-parse --git-dir)" 2>/dev/null && pwd -P)
GIT_COMMON=$(cd "$(git rev-parse --git-common-dir)" 2>/dev/null && pwd -P)
BRANCH=$(git branch --show-current)
```

**精妙之处**：
- 同一段 bash 在 `using-git-worktrees` 和 `finishing-a-development-branch` 都用
- 平台无关——bash + git 命令在所有平台都跑
- **跨平台一致性**通过共享的 git 原语

**这是"平台无关的接口"**。

### 决策 6：Codex App 沙箱降级方案

```text
当沙箱阻止 branch/push 操作时，代理提交所有工作并通知用户使用 App 的原生控件
```

**两种降级选项**：
- "Create branch"：通过 App UI 完成
- "Hand off to local"：转交本地 checkout

**关键洞察**：
- **不强行突破沙箱**——尊重环境限制
- 代理**仍然可以**跑测试、暂存文件、生成建议
- **用户**用 App 完成最终操作

**这是"沙箱下的优雅降级"**——把"做不到"变成"做能做的"。

### 决策 7：代理输出可复制的内容

```text
代理可以输出建议的分支名、commit message 和 PR 描述供用户复制
```

**关键洞察**：
- 代理**不是无能为力**——它可以"准备材料"
- 用户**复制粘贴**完成最终操作
- 保持**工作流连续性**

**这是"协作而非替代"原则**。

## 五、缺点与风险

### 1. 工具映射可能不完整

只列了 9 个工具，但 Claude Code 有更多（WebSearch、WebFetch、EnterPlanMode 等）。

**缺失**：完整工具映射。

### 2. 没有"工具差异"的说明

例如 `Read` → Codex 的 native file tools，**但语义完全一样吗**？

**缺失**：工具**语义差异**的说明。

### 3. 沙箱降级不全面

只列了"branch/push" 沙箱失败。还有其他可能的沙箱限制（test、build、network）吗？

**缺失**：完整沙箱场景的降级方案。

### 4. Legacy 说明只覆盖 `wait` 名字冲突

```text
`wait` 名称现在属于 code-mode 的 `exec/wait`
```

**问题**：其他工具名可能也有类似冲突。

**缺失**：完整的"重命名历史"。

### 5. 缺"功能差异"的覆盖

Codex 的 `update_plan` 和 Claude Code 的 `TodoWrite` 是**等价**还是有**功能差异**？

**缺失**：功能差异对比。

### 6. 缺"使用频率"统计

哪些映射**最常用**？哪些是**边缘**？

**缺失**：使用统计。

### 7. 缺"性能差异"

Codex 的 spawn_agent vs Claude Code 的 Task——**性能一样**吗？

**缺失**：性能基准。

## 六、关键洞察

### 洞察 1：这是 Superpowers 跨平台战略的"运行时翻译层"

**核心战略**：
- 技能用 **Claude Code 工具名**为标准
- 每个平台提供**映射文档**
- 翻译在**运行**时进行（不是技能编写时）

**精妙之处**：
- 不用为每个平台重写技能
- 维护成本集中在**映射表**
- 新平台只需加映射文档

**这是"接口抽象 + 平台适配"模式**。

### 洞察 2：i18n 模式应用到工具

**类比**：
- i18n：源代码用英语，运行时翻译为日语、中文等
- 工具映射：技能用 Claude Code 名，运行时翻译为 Codex 名

**相同模式**：**标准 + 适配器**。

### 洞察 3：多代理 opt-in = "性能 vs 能力"的显式权衡

```toml
[features]
multi_agent = true
```

**关键洞察**：
- Codex 默认**不开** multi_agent——节省资源
- 显式 opt-in = **用户明确选择**
- 防止"暗箱"性能开销

**这是"资源意识的配置设计"**。

### 洞察 4：Legacy 说明是"工程文档的成熟度"

```text
旧版说明：rust-v0.115.0 之前的 Codex 构建...
```

**关键洞察**：
- 不假设用户都升级
- 给旧版用户留路径
- **跨版本兼容**

**这是"工程实践的成熟度"**。

### 洞察 5：环境检测代码复用是"DRY"原则

```bash
GIT_DIR=...
GIT_COMMON=...
```

**关键洞察**：
- 同一段代码在多个技能中复用
- 平台无关（bash + git）
- **跨平台一致性**

**这是"共享接口"原则**。

### 洞察 6：Codex App 降级是"优雅失败"

```text
当沙箱阻止... 代理提交所有工作并通知用户使用 App 的原生控件
```

**关键洞察**：
- 不强行突破沙箱
- 代理**仍然有用**——跑测试、暂存、生成建议
- 用户**复制粘贴**完成最终操作

**这是"协作而非替代"原则**。

### 洞察 7：代理输出可复制材料 = "工作流连续性"

**精妙之处**：
- 代理不"放弃"——**继续提供价值**
- 用户**复制粘贴**完成最后一步
- **工作流不中断**

**这是"不完美情境下的价值延续"**。

## 九、给读者的启示

### 如果你在 Codex 上用 Superpowers
- **必做**：在 `~/.codex/config.toml` 启用 `multi_agent = true`
- **遇到 `Task` / `TodoWrite`**：查表找 Codex 等价物
- **沙箱限制**：用"Create branch"或"Hand off to local"降级
- **环境检测**：直接复用文档的 bash 代码

### 如果你写新技能
- **始终用 Claude Code 工具名**——不要写平台特定工具
- 依赖**映射文档**完成跨平台

### 如果你扩展到新平台
- **模仿本文件的结构**：
  1. 工具映射表
  2. 启用配置

  4. 环境检测
