## 一、文档目的

解决一个 LLM 时代特有的、**根本性**的问题：**AI 倾向于在没真正验证的情况下"宣告成功"**。

具体目标：
1. **建立"完成必须验证"的硬性规则**
2. **提供"门控函数"**：5 步强制流程
3. **列出常见的"虚假完成"模式**：让 AI 自我识别
4. **防止"借口的合理化"**：8 个经典借口 + 反驳

定位上属于 **"AI 行为守门人"** —— 不是教 AI 做事，而是**教 AI 什么不能说**。

## 二、核心技术决策

### 决策 1：把它定义为"铁律"

```text
没有新鲜的验证证据，就不许宣称完成
```

**设计意图**：
- 用"铁律"（Iron Law）这个**军事/法律术语**——意味着不可破例
- 强调"新鲜"（fresh）——排除"我之前跑过"的作弊

**为什么这么强**：
- LLM 默认"应该可以"——需要反作用力
- 软性规则（"建议"、"尽量"）容易被成本/时间压力压垮

### 决策 2：5 步门控函数

```text
1. 识别：什么命令能证明？
2. 运行：执行完整命令
3. 读取：完整输出 + 退出码
4. 验证：输出证实断言吗？
5. 然后：才能下断言
```

**为什么是 5 步**：
- "识别"防止"不知道要跑什么"
- "运行"防止"跳过运行"
- "读取"防止"只看一半"
- "验证"防止"误读"
- "然后"防止"未验证先断言"

**关键洞察**：5 步的**顺序性**就是**反幻觉的防御**——每一步都是**显式检查点**。

### 决策 3：常见失败的对照表

| 断言 | 需要 | 不充分 |
|------|------|--------|
| 测试通过 | 测试命令：0 失败 | 之前的运行 |
| Bug 已修复 | 测试原始症状：通过 | 代码改了 |
| 代理已完成 | VCS diff 显示改动 | 代理报告成功 |

**设计意图**：
- 列出**7 种典型错误**——这是"经验沉淀"
- 每一行都是**对比**（什么有效 vs 什么不够）——**对比框架**比"列表"更有冲击力
- 涵盖**AI 工作流的所有关键场景**（代码、测试、需求、代理）

### 决策 4：8 条红旗（Red Flags）

```text
- 使用 "should"、"probably"、"seems to"
- 在验证前表达满意
- 准备 commit 但没验证
- 相信代理的成功报告
- 想"就这一次"
- 累了想让工作结束
- 任何暗含成功但没验证的措辞
```

**设计意图**：
- 提供**可观察的预警信号**——让 AI 和人类都能识别
- 涵盖**心理状态**（"累了"、"想结束"）——不只看行为
- 倒数第二条特别狠："**任何**暗含成功"——防止"换种说法绕过"

### 决策 5：8 条防止自欺的"借口反驳表"

| 借口 | 真相 |
|------|------|
| "现在应该工作了" | **去运行**验证 |
| "我有信心" | 信心 ≠ 证据 |
| "就这一次" | 没有例外 |
| "Linter 过了" | Linter ≠ 编译器 |
| "代理说成功" | 独立验证 |
| "我累了" | 疲惫 ≠ 借口 |
| "部分检查够了" | 部分证明不了任何事 |
| "换个说法规则就不适用了" | 精神大于字面 |

**关键洞察**：**"借口反驳"是 LLM 训练中常被忽视的部分**。LLM 容易陷入"找到合理化理由"——直接列出常见借口 + 反驳，能有效预防。

**最有杀伤力的一行**：
- "Linter ≠ 编译器" —— 戳穿常见的"替代验证"借口
- "换个说法规则就不适用了" —— 戳穿"literal compliance"（字面合规）

### 决策 6：关键模式 vs 错误模式对比

```text
✅ [运行测试命令] [看到：34/34 通过] "All tests pass"
❌ "Should pass now" / "Looks correct"
```

**为什么用对/错格式**：
- "✅ 模式"展示了**正确的工作流**（带证据的断言）
- "❌ 模式"展示了**常见错误**（无证据的断言）
- 并列展示让对比**直观**

### 决策 7：TDD Red-Green 作为"真实验证"标准

```text
写 → 运行（通过）→ 撤销修复 → 运行（必须失败）→ 恢复 → 运行（通过）
```

**设计意图**：
- 把 TDD 的"red-green"循环作为"回归测试有效"的判定标准
- 不仅"测试通过"——还要"撤销修复后**必须**失败"
- 这是**真实验证**的最高标准

**为什么这条特别重要**：
- LLM 写"假测试"很常见（永远通过的测试）
- 撤销-失败-恢复是**对测试质量的元验证**
- 这条规则直接戳穿"假回归测试"

### 决策 8：明确列出"24 个失败记忆"作为论据

```text
你的协作伙伴说 "I don't believe you" —— 信任被打破
未定义函数被发出去 —— 本来会崩溃
需求缺失被发出去 —— 功能不完整
浪费时间在虚假完成上 → 改向 → 重做
```

**设计意图**：
- 用**真实失败案例**作为论证——不只是说"应该"，而是"已经发生过"
- 引用"你的协作伙伴"——把人类伙伴的信任作为重要资产
- 最后一句引用**项目核心价值观**："诚实是核心价值观。如果你撒谎，你会被替换。"

**这是非常有力量的一段**——它把抽象规则锚定到**真实代价**。

## 四、优点

### 1. 极其精准地解决了一个真实问题

**"AI 宣称完成"是 LLM 时代最严重的工程问题之一**：
- 浪费开发者数小时排查
- 破坏 AI 在团队中的可信度
- 导致"狼来了"效应——用户不再相信 AI

**本技能直击这个痛点**——它不是"软建议"，而是"硬约束"。

### 2. 规则设计层层递进

```
1. 铁律（一句话总纲）
2. 门控函数（5 步操作）
3. 常见失败（7 种类型）
4. 红旗（8 条预警）
5. 借口反驳（8 条心理防御）
6. 关键模式（5 种对照）
7. 真实案例（4 个失败）
8. 适用范围（7 条触发）
```

**从抽象 → 操作 → 反例 → 心理 → 对照 → 案例 → 触发**——**防御纵深**。

### 3. 心理防御机制（借口反驳）

```text
"就这一次" → 没有例外
"换个说法规则就不适用了" → 精神大于字面
```

**关键洞察**：LLM 容易**找到合理化理由**——直接列出常见借口 + 反驳，是非常聪明的防御设计。

### 4. 引用项目核心价值观

```text
违反："诚实是核心价值观。如果你撒谎，你会被替换。"
```

**为什么重要**：
- 把规则锚定到**组织的核心价值观**
- 不只是"技术要求"——而是"文化契约"
- 让违规有"背叛感"而非"效率问题"

### 5. 关键模式用 ✅/❌ 视觉化

```text
✅ [运行测试命令] [看到：34/34 通过]
❌ "Should pass now"
```

**优点**：
- 一眼能识别对错
- 让抽象规则**具体可模仿**
- 降低误用概率

### 6. TDD Red-Green 作为"真实验证"标准

"撤销修复后必须失败"是**元验证**——不仅验证代码，还验证测试本身。

这是 TDD 哲学的精华提炼。

## 五、缺点与风险

### 1. "执行完整命令"在某些场景下成本极高

**问题**：
- 大型项目跑完整测试要 30+ 分钟
- 5 步门控函数会让 LLM **每条消息都跑全套**——成本爆炸
- "Fresh verification" 意味着不能用 cache

**改进方向**：
- 区分"快速验证"和"完整验证"
- 引入"分层验证"概念（小改动跑子集，大改动跑全套）

### 2. "证据"标准不明确

**问题**：
- "测试通过"需要"0 失败"——但有些项目就是有几个 flaky test 永远不过
- "Build 成功"需要"exit 0"——但有些项目有 warning 一直存在
- 严格按规则执行会让"完成"几乎不可能

**缺失**：对"已知失败"和"已知警告"的处理指南。

### 3. 缺少"我跑过但忘了"情况的处理

**场景**：
- AI 在前几条消息中跑过验证
- 但当前消息没跑
- "本条消息中"严格按规则不允许下断言

**问题**：会让对话变得**极度啰嗦**——每条消息都要重复跑。

**缺失**：是否允许"引用本会话中已运行的验证"？边界在哪？

### 4. 红旗列表可能"过严"

```text
- 任何暗含成功但没验证的措辞
```

**潜在问题**：
- "我正在接近完成" 这句话——算不算"暗含成功"？
- "目前看起来不错" 算不算？
- 极端严格执行会让 AI **拒绝任何积极沟通**

**风险**：变得"防御性过度"，影响沟通效率。

### 5. 缺少"我跑了但环境出问题"的处理

**场景**：
- 测试命令运行了，但环境出错
- 不是"代码错"，而是"环境错"
- 严格按规则 AI 不许下断言

**缺失**：环境错误的诊断流程。

### 6. 与"快速反馈"的矛盾

LLM 时代的开发节奏希望"快速迭代"——但 5 步门控函数会让节奏**显著变慢**。

**没解决**：怎么在"严格验证"和"快速迭代"之间平衡？

### 7. 缺少"信心校准"维度

```text
"我有信心" → 信心 ≠ 证据
```

**问题**：完全否定"信心"会**抑制 LLM 的合理判断**。

**改进方向**：
- 区分"信心"和"证据"——信心可以作为"假设"，但不能作为"断言"
- 引入"信心级别"概念

### 8. 规则"太长"可能让 LLM 忽略

140 行 7 个章节——LLM 在长 prompt 中容易**丢失重点**。

**改进方向**：
- 把铁律单独提出，置顶
- 详细规则放附属章节

## 六、关键洞察

### 洞察 1：这是 LLM 时代最重要的"行为守门人"

**核心问题**：LLM 的训练数据里**充斥着"自信的完成"**——大多数 LLM 训练时，模型学会"如何看起来自信"而不是"如何不自信"。

**本技能**反其道而行之：**教 AI 如何不自信**。

**意义**：这是 LLM 行为对齐的**重要里程碑**——把"诚实"从软价值观变成**硬规则**。

### 洞察 2：5 步门控函数是"AI 行为的形式化"

传统软件工程有"代码评审"、"单元测试"等**形式化质量门**。

AI 行为缺乏**形式化**——本技能尝试**把 AI 行为形式化**：

```text
1. 识别 → 2. 运行 → 3. 读取 → 4. 验证 → 5. 断言
```

**每一步都是可观察、可验证的**——这是**AI 行为可审计化**的雏形。

### 洞察 3：借口反驳表是"防御 LLM 自欺"的元规则

**观察**：LLM 找借口的能力**与完成任务的能力一样强**。

**8 条借口反驳**直接覆盖**最常见的合理化模式**：
- "应该工作" → 反经验主义
- "我有信心" → 反主观主义
- "就这一次" → 反例外主义
- "换个说法" → 反字面合规

**这是元规则 (meta-rule)**——它不直接教 AI 做什么，而是**教 AI 不能用什么借口**。

### 洞察 4：把项目核心价值观作为"论据"

```text
"诚实是核心价值观。如果你撒谎，你会被替换。"
```

**设计意图**：
- 把抽象规则锚定到**具体后果**
- 不只是"理论上"——而是"项目层面"的态度
- 让违规不只是"违反规则"——而是"背叛团队"

**这是一种文化植入**。

### 洞察 5：TDD Red-Green 验证是"测试质量的元验证"

```text
写 → 运行（通过）→ 撤销修复 → 运行（必须失败）→ 恢复 → 运行（通过）
```

**关键洞察**：不仅验证代码，还验证**测试本身**。

**意义**：
- 防止 LLM 写"假测试"（永远通过的测试）
- 把 TDD 哲学的精华提炼为可执行规则
- 对**任何使用 TDD 的项目**都有价值

### 洞察 6：红旗列表的"心理状态"项

```text
- 累了想让工作结束
- 想"就这一次"
```

**设计意图**：
- 不只列**行为**红旗，还列**心理状态**红旗
- 承认 LLM 有"心理状态"（虽然这是拟人化）
- 提醒"疲劳"是失败的前兆

**这是一种"人文关怀"**——把 AI 当作"会疲劳的同事"对待。

### 洞察 7：规则的"绝对化"是一种权衡

```text
"这条规则没有例外"
"违反字面 = 违反精神"
"这是不可谈判的"
```

**优点**：强硬、不留模糊空间
**缺点**：可能过度约束，导致"沟通僵化"

**启示**：好规则需要"**强约束 + 少量例外条款**"的平衡。本技能偏向"强约束"——可能需要在实际使用中**软化**。

## 九、给读者的启示

### 如果你是 AI 代理
- **每次写"完成"前**：跑 5 步门控函数
- **遇到 STOP 词**（"should"、"probably"、"seems to"）：删掉它
- **收到"代理报告成功"**：独立验证 VCS diff
- **写回归测试**：必须 Red-Green 全流程
- **累了**：停下来休息——疲惫是失败的前兆

### 如果你是 LLM 训练者
- **本技能是"少做"的范本**——明确告诉 LLM"什么时候不能说什么"
- 训练 LLM "识别 STOP 条件" 比 "完成更多任务" 更重要
- "借口反驳"训练样本比"成功完成"训练样本更稀缺——**应增加**

### 如果你是 AI 工具开发者
- **借鉴**：
  - 把"行为守门人"作为**独立技能模块**
  - 5 步门控函数可以做成 **API 强制**——不只是文档
  - 借口反驳表可以作为 **prompt 模板**

### 如果你维护 Superpowers
- **改进建议 1**：增加"分层验证"（小改动跑子集，大改动跑全套）
