## 一、文档目的

解决一个具体的工程问题：**当 AI 协调者面临 2+ 个独立任务时，怎么最大化吞吐量？**

具体目标：
1. **定义触发条件**：什么时候适合并行（什么时候不适合）
2. **提供决策图**：用 Graphviz 可视化"独立 + 可并行"的判断
3. **教 Prompt 编写**：4 段式结构（聚焦、自洽、约束、明确输出）
4. **真实案例示范**：6 个失败 → 3 个并行代理 → 0 冲突

定位上属于 **"扇出模式"（fan-out pattern）的工程规范**——把"分而治之"的古老思想**在 AI 时代形式化**。

## 二、核心技术决策

### 决策 1：独立性是"硬性前置条件"

```text
Multiple failures?
  → Are they independent?
    → YES → Can they work in parallel?
      → YES → Parallel dispatch
      → NO (shared state) → Sequential agents
    → NO (related) → Single agent investigates all
```

**设计意图**：
- 独立性是**第一性条件**——不独立就分不开
- 共享状态是**第二性条件**——独立但有共享也不行
- 三个**显式失败模式**被列出

**精妙之处**：决策图把"何时不用"和"何时用"都画出来——**避免误用**。

### 决策 2：代理"绝不继承父会话上下文"

```text
它们绝不应该继承你的会话上下文或历史 —— 你要构造它们所需的精确信息
```

**关键洞察**：
- 子代理的"上下文继承"看似"贴心"，实际上是**噪声**——大量无关历史会干扰子代理
- 协调者的职责是**精心构造子代理的最小上下文**
- 这同时**节省 token**（不传无关历史）

**哲学根基**：Unix 的 "small tools, compose" —— 每个代理都是独立小工具。

### 决策 3：4 段式 Prompt 结构

```
1. 聚焦的（Focused）— 一个明确的问题域
2. 自洽的（Self-contained）— 全部所需上下文
3. 明确指定输出的（Specific about output）— 返回什么？
```

**缺失第 4 段**：**约束**（Constraints）——不只"做什么"，还要"不做什么"。

实际上文档后续的 "Common Mistakes" 章节专门讲了 4 类常见错误：
- 太宽泛（无聚焦）
- 无上下文（无自洽）
- 无约束
- 输出含糊

**设计意图**：通过"正反对比"教 Prompt 编写。

### 决策 4：约束条件（Constraints）的硬性

```text
Do NOT just increase timeouts - find the real issue.
```

**为什么"不要" 这么重要**：
- LLM 倾向于"最小阻力路径"——加 timeout 看起来能解决
- 显式禁止 = 切断这条 LLM 偏好
- 引导 LLM 走"真正的根因分析"

**这是反 LLM 捷径偏好的范例**。

### 决策 5：明确"分派 4 步流程"

```text
1. 识别独立域（Identify Independent Domains）
2. 创建聚焦的代理任务（Create Focused Agent Tasks）
3. 并行分派（Dispatch in Parallel）
4. 审查与集成（Review and Integrate）
```

**4 步覆盖了完整生命周期**：
- 前：分析
- 中：分派
- 后：集成

**关键洞察**：**第 4 步"审查与集成"最容易被忽略**——很多人以为"分出去就不用管了"，实际上**整合阶段风险最高**（冲突、遗漏、系统性错误）。

### 决策 6：决策图用 Graphviz 渲染

```dot
digraph when_to_use {
    "Multiple failures?" [shape=diamond];
    ...
}
```

**为什么用图**：
- "独立/相关"、"并行/串行"这些判断**用树状结构难表达**
- 决策图（digraph）是**形式化决策树**的标准方式
- 视觉上比"if-else 列表"更清晰

**技术细节**：
- `shape=diamond` 是决策节点
- `shape=box` 是动作节点
- 边上的 `label` 是条件

### 决策 7：禁止"修一个可能修其他"的串行

```text
相关失败：修一个可能修其他 —— 先一起调查
```

**为什么这么严**：
- 如果 3 个失败同源，并行修可能产生 3 个冲突的修复
- 串行调查能识别共同根因
- **这是"避免并行浪费"的防御**

**反面教训**：很多团队"为了并行而并行"——结果产生 5 个冲突的 PR。

### 决策 8：完整真实案例

```text
场景：6 个失败跨 3 个文件
分派：3 个并行代理
结果：3 个修复零冲突
```

**价值**：
- 证明"独立域"判断的可行性
- 展示"3 个代理改 3 个不同文件"的成功模式
- 提供"时间节省"的具体数字

**这比"理论说应该并行"更有说服力**。

## 四、优点

### 1. 决策图（digraph）极其实用

```dot
"Multiple failures?" -> "Are they independent?" [label="yes"];
```

**优点**：
- 视觉化决策树比纯文字更清晰
- Graphviz 是**生态标准**——可以生成 PNG/SVG
- 提供了**形式化决策**的范例

### 2. 明确禁止"上下文继承"

```text
它们绝不应该继承你的会话上下文或历史
```

**优点**：
- 直接戳穿"上下文共享 = 贴心"的迷思
- 节省 token（不传无关历史）
- 让子代理"专注"——不被无关信息干扰

**这是反直觉但正确的设计**。

### 3. Prompt 结构 3 大要素 + 4 类常见错误

**3 大要素**（正面）：聚焦、自洽、明确输出
**4 类错误**（反面）：太宽泛、无上下文、无约束、含糊输出

**对比教学**比"单一列表"更有效。

### 4. 真实案例 + 实际影响

```text
2025-10-03 调试会话
6 个失败跨 3 个文件
3 个并行代理
零冲突
```

**优点**：
- 不只是"理论上可行"——有**真实数据**支持
- "时间节省"具体可感
- 增加了文档的**可信度**

### 5. 集成阶段风险明确化

```text
审查每个摘要
检查冲突
跑完整套件
抽查（代理可能犯系统性错误）
```

**关键洞察**：
- "代理可能犯系统性错误"——**承认 LLM 不是完美的**
- "跑完整套件"——**集成阶段需要完整验证**
- 集成阶段**和分派阶段同样重要**

### 6. 明确的"不适用"清单

```text
相关失败
需要完整上下文
探索性调试
共享状态
```

**优点**：
- 主动暴露**反模式**——防止误用
- 4 个反模式覆盖了**典型误用场景**

## 五、缺点与风险

### 1. 决策图用 Graphviz 渲染，但渲染结果缺失

**问题**：
- 文档给的是 `.dot` 源码
- 实际渲染需要 `dot` 命令或在线渲染器
- 在 Markdown 预览中看不到图

**改进**：
- 提供预渲染的 PNG/SVG
- 或用 mermaid（GitHub 原生支持）

### 2. "独立"判断标准不明确

**问题**：
- "独立性"是核心判断，但怎么判断？
- 文件不同就独立？模块不同就独立？
- 共享数据库表算"共享状态"吗？

**缺失**：**独立性的客观判定标准**。

### 3. 缺失败案例

**问题**：
- 1 个真实成功案例
- 0 个失败案例
- "什么时候并行会失败"没说

**改进**：增加"哪些情况并行会出问题"的案例。

### 4. 整合阶段的"冲突检测"没说怎么做

**问题**：
- "检查冲突"——怎么检查？
- 文件级冲突（同一文件不同行）？
- 语义级冲突（不同文件但逻辑冲突）？

**缺失**：**冲突检测的具体方法**。

### 5. "代理可能犯系统性错误"没说怎么"系统性"

**问题**：
- 一次性错误容易抓
- 系统性错误（如总是用某种反模式）怎么识别？

**缺失**：系统性错误的识别模式。

### 6. 没有"成本-收益"分析

**问题**：
- 并行 3 个代理 = 3x token 消耗
- 是否值得？
- 什么任务"值 3x"？

**缺失**：**何时值得为并行付出成本**的判断。

### 7. 没有"代理饱和度"考虑

**问题**：
- 模型 API 有 rate limit
- 平台对子代理数量有限制（如 Claude Code 默认限制）
- "并行 10 个代理"可能根本不可行

**缺失**：代理数量上限的考虑。

### 8. 缺少"任务规模"的判断

**问题**：
- "修 1 个文件"适合并行吗？（也许单代理更快）
- "修 100 个文件"必须并行吗？
- 任务的"并行价值"有阈值

**缺失**：**任务规模 vs 并行收益**的判断。

## 六、关键洞察

### 洞察 1：这是"扇出-汇总"模式的 AI 工程化

```
        ┌─ Agent 1 (独立域 A)
协调者 ─┼─ Agent 2 (独立域 B)
        └─ Agent 3 (独立域 C)
                ↓
            整合
```

**本质**：古老的"divide and conquer" + 古老的"map-reduce"——在 AI 时代的**重新形式化**。

**为什么重要**：
- 把**分布式系统思想**应用到 AI 工作流
- 思考方式从"AI 串行做事"变成"AI 协调多代理并发"
- 这是 **AI 编排（orchestration）** 的入门

### 洞察 2：上下文隔离是"性能 + 正确性"的双重胜利

```text
代理不继承父会话上下文
```

**性能收益**：
- 子代理收到的 token 少
- 响应更快、成本更低

**正确性收益**：
- 子代理不被无关历史干扰
- 不会被"上次会话的失败"误导
- 每个代理都是**纯净起点**

**这是反直觉但重要的设计决策**。

### 洞察 3：约束是 LLM 偏好的"反作用力"

```text
Do NOT just increase timeouts - find the real issue.
```

**观察**：LLM 默认走"最小阻力路径"——加 timeout 看起来能"解决"问题。

**对策**：**显式禁止常见捷径**——用 "Do NOT" 句式。

**这是工程化 LLM 的核心模式**：**不要假设 LLM 会"做正确的事"**——要**显式禁止错误路径**。

### 洞察 4：集成阶段 = 风险最高阶段

```text
分派：低风险（独立域）
执行：低风险（隔离上下文）
整合：高风险（冲突、遗漏、系统性错误）
```

**传统软件工程**："整合"是"提交 PR 之后"——晚暴露风险。
**本技能**："整合"是**协调者的明确职责**——早发现冲突。

**反模式**：以为"分出去就不用管"——会积累大量冲突。

### 洞察 5：决策图是"形式化决策"的范例

```dot
"Multiple failures?" -> "Are they independent?" [label="yes"];
```

**形式化决策的优势**：
- 可以**测试**（遍历所有路径）
- 可以**讨论**（不依赖文字描述）
- 可以**演化**（加节点而不改全文）

**对比**：纯文字的"if A 则 B，否则 C"——易遗漏、难维护。

### 洞察 6：4 类常见错误的"反例教学"

```text
❌ "Fix all the tests"
✅ "Fix agent-tool-abort.test.ts"
```

**反例教学的优势**：
- 让 LLM 看到**具体错误模式**
- 触发**模式识别**——比抽象规则更有效
- 让"反 LLM 捷径"具体化

### 洞察 7：成本-收益未明说但隐含

```text
3 个问题在 1 个问题的时间里解决
```

**暗示**：3x 速度 = 3x 成本？不一定——
- 子代理 token 消耗可能比主代理低
- 并行 = 真实墙钟时间大幅缩短
- 收益/成本比**对独立小任务特别高**

**缺失**：文档没明示何时"成本不值得"——需要读者自己判断。

## 九、给读者的启示

### 如果你是 AI 协调者
- **遇到 3+ 独立失败**：考虑并行分派
- **每个子代理**：构造最小独立上下文
- **Prompt 必含**：聚焦、自洽、约束、明确输出
- **整合阶段**：必跑完整套件

### 如果你是 LLM 训练者
- **本技能展示了"扇出-汇总"模式**——训练数据应包含
- **约束的显式禁止**（"Do NOT..."）是反 LLM 捷径的关键
- **决策图形式化**——训练时重视"分支判断"

### 如果你是 AI 工具开发者
- **借鉴**：
  - "代理不继承父上下文"作为 API 默认值
  - "约束"作为 prompt 模板必含字段
  - "冲突检测"作为集成阶段必跑步骤

### 如果你维护 Superpowers
- **改进建议 1**：补充"独立性的客观判断标准"（如"文件级独立 = 改不同文件"）
- **改进建议 2**：增加"何时不值得并行"的成本-收益分析
- **改进建议 3**：增加"冲突检测"的具体方法（git diff、文件级、语义级）
- **改进建议 4**：提供预渲染的决策图 PNG/SVG
- **改进建议 5**：增加"失败案例"——哪些情况并行会出问题

### 如果你在做 AI 团队管理
- **本技能可作为"AI 子代理使用规范"的核心条款**
- 强调"上下文隔离"——避免把 AI 当成"共享大脑"
- 强调"约束必含"——不要让 LLM 走捷径
