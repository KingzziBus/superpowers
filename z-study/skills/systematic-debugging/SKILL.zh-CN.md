# 系统化调试

## 概述

随机修复浪费时间并产生新 bug。快速补丁掩盖潜在问题。

**核心原则：** 永远在尝试修复前找到根因。症状修复是失败。

**违反此过程的字面就是违反调试的精神。**

## 铁律

```
没有根因调查之前，不进行修复
```

如果你没有完成 Phase 1，你就不能提出修复。

## 何时使用

用于任何技术问题：
- 测试失败
- 生产环境 bug
- 意外行为
- 性能问题
- 构建失败
- 集成问题

**特别在以下情况使用：**
- 在时间压力下（紧急情况让人倾向于猜测）
- "就一个快速修复" 似乎很明显
- 你已经尝试了多个修复
- 之前的修复不起作用
- 你没有完全理解这个问题

**不要在以下情况跳过：**
- 问题看起来简单（简单 bug 也有根因）
- 你在赶时间（仓促保证返工）
- 经理想现在修好（系统化比胡乱尝试更快）

## 四个阶段

你必须在进入下一阶段前完成每一阶段。

### Phase 1：根因调查

**在任何修复之前：**

1. **仔细阅读错误信息**
   - 不要跳过错误或警告
   - 它们通常包含精确的解决方案
   - 完整阅读堆栈跟踪
   - 注意行号、文件路径、错误码

2. **一致地复现**
   - 你能可靠地触发它吗？
   - 确切的步骤是什么？
   - 每次都发生吗？
   - 如果不能复现 → 收集更多数据，不要猜测

3. **检查最近的更改**
   - 什么变化可能导致这个？
   - Git diff、近期提交
   - 新依赖、配置变更
   - 环境差异

4. **在多组件系统中收集证据**

   **当系统有多个组件时（CI → 构建 → 签名，API → 服务 → 数据库）：**

   **在提出修复之前，添加诊断工具：**
   ```
   对于每个组件边界：
     - 记录什么数据进入组件
     - 记录什么数据退出组件
     - 验证环境/配置传播
     - 检查每层的状态

   运行一次以收集证据显示哪里破坏
   然后分析证据以识别失败的组件
   然后调查该特定组件
   ```

   **示例（多层系统）：**
   ```bash
   # 第 1 层：工作流
   echo "=== 工作流中可用的 secrets：==="
   echo "IDENTITY: ${IDENTITY:+SET}${IDENTITY:-UNSET}"

   # 第 2 层：构建脚本
   echo "=== 构建脚本中的环境变量：==="
   env | grep IDENTITY || echo "IDENTITY 不在环境中"

   # 第 3 层：签名脚本
   echo "=== Keychain 状态：==="
   security list-keychains
   security find-identity -v

   # 第 4 层：实际签名
   codesign --sign "$IDENTITY" --verbose=4 "$APP"
   ```

   **这揭示：** 哪一层失败（secrets → 工作流 ✓，工作流 → 构建 ✗）

5. **追踪数据流**

   **当错误在调用栈深处时：**

   参见本目录中的 `root-cause-tracing.md` 了解完整的反向追踪技术。

   **快速版本：**
   - 不良值的起源在哪？
   - 谁用不良值调用了这个？
   - 继续向上追踪直到找到源头
   - 在源头修复，而非症状

### Phase 2：模式分析

**在修复前找到模式：**

1. **找到工作中的示例**
   - 在同一代码库中找到类似的工作代码
   - 什么工作的东西与坏掉的类似？

2. **对比参考**
   - 如果在实现模式，完整阅读参考实现
   - 不要略读 —— 逐行阅读
   - 在应用前完全理解模式

3. **识别差异**
   - 工作和坏掉之间的区别是什么？
   - 列出每个差异，无论多小
   - 不要假设"那不可能有关系"

4. **理解依赖**
   - 这需要哪些其他组件？
   - 什么设置、配置、环境？
   - 它做了什么假设？

### Phase 3：假设与测试

**科学方法：**

1. **形成单一假设**
   - 明确陈述："我认为 X 是根因，因为 Y"
   - 写下来
   - 具体而非模糊

2. **最小化测试**
   - 做最小的可能变更来测试假设
   - 一次一个变量
   - 不要一次修多个东西

3. **在继续前验证**
   - 有效吗？是 → Phase 4
   - 无效？形成新假设
   - 不要在上面添加更多修复

4. **当你不知道时**
   - 说"我不理解 X"
   - 不要假装知道
   - 寻求帮助
   - 研究更多

### Phase 4：实施

**修复根因，而非症状：**

1. **创建失败测试用例**
   - 最简单的复现
   - 如果可能，自动化测试
   - 如果没有框架，临时测试脚本
   - 必须在修复前
   - 使用 `superpowers:test-driven-development` 技能写适当的失败测试

2. **实施单一修复**
   - 处理识别出的根因
   - 一次一个变更
   - 没有"既然来了" 改进
   - 没有捆绑重构

3. **验证修复**
   - 测试现在通过了吗？
   - 没有破坏其他测试？
   - 问题实际解决了吗？

4. **如果修复不起作用**
   - 停下来
   - 数：你已经尝试了多少个修复？
   - 如果 < 3：返回 Phase 1，用新信息重新分析
   - **如果 ≥ 3：停下来质疑架构（见下面 step 5）**
   - 不要在没有架构讨论的情况下尝试第 4 个修复

5. **如果 3+ 修复失败：质疑架构**

   **指示架构问题的模式：**
   - 每个修复在不同地方揭示新的共享状态/耦合/问题
   - 修复需要"大规模重构" 来实施
   - 每个修复在其他地方产生新症状

   **停下来质疑基础：**
   - 这个模式从根本上是合理的吗？
   - 我们是在"纯粹靠惯性坚持下去"吗？
   - 我们应该重构架构还是继续修复症状？

   **在尝试更多修复前与你的 human partner 讨论**

   这不是失败的假设 —— 这是错误的架构。

## 红旗 - 停下来遵循流程

如果你发现自己在想：
- "快速修复先，之后调查"
- "试试改 X 看看是否工作"
- "添加多个变更，跑测试"
- "跳过测试，我会手动验证"
- "可能是 X，让我修复"
- "我不完全理解但这可能工作"
- "模式说 X 但我会不同地适应它"
- "这里是主要问题：[列出未调查的修复]"
- 在追踪数据流前提出解决方案
- **"再来一次修复尝试"（已经尝试 2+ 次时）**
- **每个修复在不同地方揭示新问题**

**所有这些都意味着：停下来。返回 Phase 1。**

**如果 3+ 修复失败：** 质疑架构（见 Phase 4.5）

## 你的 human partner 的"你做错了" 信号

**注意以下重定向：**
- "那不是正在发生吗？" - 你没验证就假设了
- "它会显示……吗？" - 你应该添加了证据收集
- "停止猜测" - 你在不理解的情况下提出修复
- "深入思考这个" - 质疑基础，而不仅仅是症状
- "我们卡住了？"（沮丧） - 你的方法不起作用

**当你看到这些：** 停下来。返回 Phase 1。

## 常见合理化

| 借口 | 现实 |
|--------|---------|
| "问题简单，不需要流程" | 简单问题也有根因。对简单 bug 流程是快的。 |
| "紧急，没时间走流程" | 系统化调试比猜测-检查-折腾更快。 |
| "先试试这个，然后调查" | 第一个修复设定模式。从一开始做对。 |
| "确认修复有效后再写测试" | 未测试的修复不会持久。先测试证明它。 |
| "一次多个修复节省时间" | 无法隔离哪个有效。产生新 bug。 |
| "参考太长，我会适应模式" | 部分理解保证 bug。完整阅读。 |
| "我看到问题了，让我修复" | 看到症状 ≠ 理解根因。 |
| "再来一次修复尝试"（2+ 失败后） | 3+ 失败 = 架构问题。质疑模式，不要再修。 |

## 快速参考

| 阶段 | 关键活动 | 成功标准 |
|-------|---------------|------------------|
| **1. 根因** | 读错误、复现、检查变化、收集证据 | 理解 WHAT 和 WHY |
| **2. 模式** | 找到工作中的示例、对比 | 识别差异 |
| **3. 假设** | 形成理论、最小化测试 | 确认或新假设 |
| **4. 实施** | 创建测试、修复、验证 | Bug 解决，测试通过 |

## 当流程揭示"没有根因"时

如果系统化调查揭示问题确实是环境性的、依赖时间的或外部的：

1. 你已经完成了流程
2. 记录你调查的内容
3. 实施适当的处理（重试、超时、错误消息）
4. 添加监控/日志以供未来调查

**但是：** 95% 的"没有根因" 案例是不完整的调查。

## 支持技术

这些技术是系统化调试的一部分，可在本目录中获得：

- **`root-cause-tracing.md`** - 通过调用栈反向追踪 bug 找到原始触发器
- **`defense-in-depth.md`** - 找到根因后在多层添加验证
- **`condition-based-waiting.md`** - 用条件轮询替换任意超时

**相关技能：**
- **superpowers:test-driven-development** - 用于创建失败测试用例（Phase 4, Step 1）
- **superpowers:verification-before-completion** - 在声称成功前验证修复有效

## 现实影响

来自调试会话：
- 系统化方法：15-30 分钟修复
- 随机修复方法：2-3 小时的折腾
- 首次修复率：95% vs 40%
- 引入的新 bug：接近零 vs 常见
