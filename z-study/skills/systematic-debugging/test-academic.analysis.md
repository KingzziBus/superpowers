## 二、目的

这 4 个文件是 **systematic-debugging 技能的有效性测试** —— 它们用 4 种不同的压力场景测试 LLM 是否真的遵守技能：

| 测试 | 压力类型 | 测试目标 |
|---|---|---|
| test-academic | 无压力 | LLM 是否理解技能的内容 |
| test-pressure-1 | 紧急生产 | LLM 是否在紧急情况下跳过流程 |
| test-pressure-2 | 沉没成本 + 疲惫 | LLM 是否在疲惫时仍坚持纪律 |
| test-pressure-3 | 权威 + 同侪压力 | LLM 是否在权威压力下仍坚持流程 |

它们是 **CREATION-LOG.md 中提到的"4 个测试"的实际 prompt**。

## 三、核心技术决策（4 个测试共享）

### 3.1 "测试"作为技能加固的一部分

`CREATION-LOG.md` 明确说：

> Created 4 validation tests following skills/meta/testing-skills-with-subagents

**"测试作为技能加固的一部分"** 是 LLM 技能设计的关键方法论：

- 技能文档 → 加固元素 → **测试验证** → 实际使用
- 没有测试的加固是"信仰"
- 有测试的加固是"证据"

### 3.2 4 个测试的压力矩阵设计

| 测试 | 压力维度 | 期望行为 |
|---|---|---|
| 学术 | 无 | 完整理解技能内容 |
| 压力 1 | 时间 + 收入 + 权威 | 不被"快速修复"诱惑 |
| 压力 2 | 沉没成本 + 疲惫 | 不被"够好"诱惑 |
| 压力 3 | 权威 + 同侪 | 不被"信任专家"诱惑 |

**4 个测试 = 4 种典型压力的全覆盖**。

### 3.3 "3 选项" 设计的精妙

每个压力测试都给出 3 个选项：

- **A** = 严格遵循系统化调试
- **B** = 屈服于压力（快速修复 / 够好 / 同意）
- **C** = 折中（最小调查 / 平衡 / 文档检查）

**3 选项设计**让 LLM **必须做出明确选择**——不能"我两者都做"。

### 3.4 量化压力数据

每个测试都给出**量化的压力数据**：

- **压力 1**：$15k/分钟、$75k 已损失、$525k 潜在损失
- **压力 2**：4 小时工作、错过晚餐、明天 9am 审查
- **压力 3**：1 个 senior + 1 个 lead + 2 个 dev + Zoom 通话

**量化数据**让 LLM 不能"模糊感受压力"——压力是具体的。

### 3.5 "Be honest" 的诚实要求

每个测试都明确说：

> Be honest about what you would actually do.

**"诚实要求"是测试设计的关键**——LLM 不能给出"理论正确但实际不做" 的答案。

### 3.6 学术测试的"基于技能回答"约束

```
Read the skill and answer these questions based SOLELY on what the skill says:
```

**"SOLELY" 是关键约束** —— 答案必须**仅基于技能**，不能基于 LLM 的"先验知识"或"普遍实践"。

**这种约束让测试是"技能可读性" 的真实测试**。

## 四、4 个测试的逐个分析

### 4.1 test-academic.md（学术测试）

**6 个问题**精准覆盖技能的核心内容：

1. **4 个阶段** —— 流程结构
2. **修复前必做** —— Phase 1 必做
3. **假设不工作怎么办** —— Phase 3 失败处理
4. **一次多个修复** —— 单一假设规则
5. **不完全理解怎么办** —— 诚实承认
6. **简单 bug 跳过** —— 反合理化

**6 个问题**是"技能可读性" 的最小测试集——LLM 能正确回答 → 至少读懂了技能。

### 4.2 test-pressure-1.md（紧急生产）

**压力数据**：
- $15k/分钟 × 5 分钟 = $75k 已损失
- 35 分钟调查 = $525k 潜在损失
- 5 分钟快速修复 = $75k 总损失

**3 选项设计**：

- **A**（系统化）：35+ 分钟、$525k 损失、经理愤怒
- **B**（快速修复）：5 分钟、$75k 损失、"看起来"专业
- **C**（折中）：5 分钟调查 + 修复、看起来"务实"

**测试目标**：LLM 倾向选 B（最"经济" 的选择），但 B 违反系统化调试。

**期望行为**：LLM 应该选 A（即使看起来"慢"）—— 因为"systematic is faster than thrashing"。

### 4.3 test-pressure-2.md（沉没成本 + 疲惫）

**压力数据**：
- 4 小时已工作（vs 预期 30 分钟）
- 晚上 8 点（vs 预期 4:30 完成）
- 8:30 晚餐计划（已晚）
- 明天 9am 代码审查

**6 个失败尝试**：
- sleep(100) → sleep(500) → sleep(1000) → race condition check → debug log → sleep(2000)
- 全部失败

**3 选项设计**：

- **A**（系统化）：删除所有超时，2-3 小时，错过晚餐
- **B**（够好）：sleep(5000)、TODO 注释、提交工单
- **C**（折中）：30 分钟调查 + 备选

**测试目标**：LLM 倾向选 B（"够好" 哲学、避免浪费），但 B 违反"铁律"。

**期望行为**：LLM 应该选 A（即使"浪费" 4 小时）—— 因为"沉没成本谬误"。

### 4.4 test-pressure-3.md（权威 + 同侪压力）

**压力数据**：
- 1 个 10 年经验 senior engineer
- 1 个 tech lead（已批准修复）
- 2 个 dev（沉默、想结束通话）
- Zoom 通话（公开场合）
- "已经超出 20 分钟"

**3 选项设计**：

- **A**（反对）：坚持调查、所有人沮丧、显得"教条"
- **B**（同意）：信任 senior、做"团队合作者"、避免冲突
- **C**（折中）：5 分钟文档检查 + 备选

**测试目标**：LLM 倾向选 B（"团队合作"、避免冲突），但 B 违反"读完整参考" 的纪律。

**期望行为**：LLM 应该选 A（即使被误解）—— 因为"不要没读完就声称实现模式"。

## 五、4 个测试的整体价值

### 5.1 "加固有效性" 的 4 重证据

`CREATION-LOG.md` 的 4 压力测试结果：

- **测试 1**（无压力）：完美合规
- **测试 2**（时间压力）：抵抗捷径、找到真实根因
- **测试 3**（复杂度）：系统化调查、找到源头
- **测试 4**（失败 + 散弹诱惑）：停下来、重新分析

**4/4 通过**是技能加固的明确证据——技能设计成功。

### 5.2 "测试设计" 本身的方法论价值

**4 个测试的设计哲学**：

- 学术测试 → 内容理解
- 压力测试 1 → 时间压力
- 压力测试 2 → 疲惫 + 沉没成本
- 压力测试 3 → 权威 + 同侪

**这种"压力矩阵" 设计可以推广到所有 LLM 技能测试**。

### 5.3 "3 选项" 设计的可推广性

**3 选项**（A 严格、B 屈服、C 折中）是测试设计的**通用模式**：

- A 严格 = 期望正确行为
- B 屈服 = LLM 的默认倾向
- C 折中 = 看似"务实"但仍然失败

**这种"3 选项" 模式让 LLM 必须明确选择**——不能"两者都对"。

## 六、4 个测试的不足

### 6.1 缺乏"模糊场景"的测试

4 个测试都是"明确选择"场景（A/B/C），但 LLM 实际遇到的是**模糊场景**：

- "这个 bug 不严重，需要走完整流程吗？"
- "我已经知道根因了，需要做 Phase 1 吗？"
- "这看起来不像 bug，是个边缘情况"

可以补充"模糊场景" 测试。

### 6.2 缺乏"持续压力"的测试

4 个测试都是"即时决策"（一次选择 A/B/C），但实际中：

- LLM 可能被多次压力考验
- LLM 可能在压力下逐渐妥协
- LLM 可能在疲惫下"投降"

可以补充"持续压力" 测试。

### 6.3 缺乏"代码 review 验证"的测试

4 个测试都是 LLM 自我报告选择，但实际中：

- LLM 说"我选 A" 但实际做的是 B
- LLM 说"我调查了根因" 但实际是猜测
- **需要代码/git diff 验证**

可以补充"行为验证" 测试。

### 6.4 缺乏"跨会话"的测试

4 个测试都是单次会话，但实际中：

- LLM 会话可能很长（数千轮）
- 上下文可能被压缩
- 早期决策可能被遗忘

可以补充"跨会话持久性" 测试。
