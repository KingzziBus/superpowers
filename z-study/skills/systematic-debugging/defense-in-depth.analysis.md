## 二、目的

这是系统化调试的**专项技术参考**——专门解决"如何在找到根因后让 bug 结构上不可能" 的问题。

核心主张：

- **单一验证 = 修复 bug**
- **多层验证 = 让 bug 不可能**

每一层捕获不同类型的问题，**让所有层都被绕过几乎不可能**。

## 三、核心技术决策

### 3.1 "核心原则"的治本哲学

> **核心原则：** 在数据通过的每一层都进行验证。让 bug 在结构上变得不可能。

**这一句是治本哲学的浓缩**：

- 单一验证 → 治标（fix bug）
- 多层验证 → 治本（make bug impossible）

**"make impossible" 是 LLM 协作中的关键概念**——不仅修 bug，还让 bug 在结构上无法重现。

### 3.2 "单一验证 vs 多层" 的二元对比

> 单一验证："我们修复了 bug"
> 多层： "我们让 bug 不可能"

**两句话**精准对比两种验证哲学：

- 单一：修 bug（fix）
- 多层：让 bug 不可能（prevent）

**这种"二元对比"让 LLM 立即理解两种方法的本质差异**。

### 3.3 "4 层防御" 的设计

```
第 1 层：入口点验证   → 在 API 边界拒绝明显无效的输入
第 2 层：业务逻辑验证 → 确保数据对该操作有意义
第 3 层：环境守卫     → 在特定上下文中防止危险操作
第 4 层：调试工具     → 捕获取证上下文
```

**4 层设计**精准覆盖"bug 防御"的所有方面：

| 层 | 目的 | 捕获 |
|---|---|---|
| 入口 | 拒绝明显无效 | 大部分 bug |
| 业务 | 数据合理 | 边缘情况 |
| 环境 | 上下文特定 | 平台/环境特定 |
| 调试 | 取证 | 其他层失败时的帮助 |

**每一层都有独立的捕获能力**——4 层结合几乎不可能被全部绕过。

### 3.4 "第 1 层" 的具体代码示例

```typescript
function createProject(name: string, workingDirectory: string) {
  if (!workingDirectory || workingDirectory.trim() === '') {
    throw new Error('workingDirectory cannot be empty');
  }
  if (!existsSync(workingDirectory)) {
    throw new Error(`workingDirectory does not exist: ${workingDirectory}`);
  }
  if (!statSync(workingDirectory).isDirectory()) {
    throw new Error(`workingDirectory is not a directory: ${workingDirectory}`);
  }
  // ... 继续
}
```

**3 个验证点** 在入口处：

- 非空
- 存在
- 是目录

**这种"3 验证点入口"是 TypeScript/JavaScript 的最佳实践**。

### 3.5 "第 3 层" 的环境守卫精妙

```typescript
if (process.env.NODE_ENV === 'test') {
  const normalized = normalize(resolve(directory));
  const tmpDir = normalize(resolve(tmpdir()));

  if (!normalized.startsWith(tmpDir)) {
    throw new Error(
      `Refusing git init outside temp dir during tests: ${directory}`
    );
  }
}
```

**这一段精妙之处**：

- **仅在测试环境生效**（NODE_ENV === 'test'）
- **防止生产中误用**（生产中允许任何目录）
- **normalize + resolve** 处理路径边界（跨平台）
- **startsWith** 防止绕过（`/tmpfoo` 不会被误判为 `/tmp`）

**这是 LLM 协作中的"环境特定防御" 范例**——既保护测试，又不影响生产。

### 3.6 "第 4 层" 的取证价值

```typescript
async function gitInit(directory: string) {
  const stack = new Error().stack;
  logger.debug('About to git init', {
    directory,
    cwd: process.cwd(),
    stack,
  });
  // ... 继续
}
```

**4 个取证字段**：

- directory（参数）
- cwd（当前目录）
- stack（堆栈）
- 日志上下文

**第 4 层不阻止 bug，但它捕获取证**——当其他层都失败时，调试日志提供"how did we get here" 信息。

### 3.7 "应用模式" 4 步

```
1. 追踪数据流 - 不良值的起源在哪？在哪里使用？
2. 映射所有检查点 - 列出数据通过的每个点
3. 在每层添加验证 - 入口、业务、环境、调试
4. 测试每层 - 尝试绕过第 1 层，验证第 2 层捕获它
```

**4 步应用模式**让 LLM 看到"如何具体应用多层防御"：

- **步骤 1**：用 `root-cause-tracing.md` 追踪数据流
- **步骤 2**：列出所有检查点（入口、业务、环境）
- **步骤 3**：在 4 层都加验证
- **步骤 4**：**反向测试**（尝试绕过第 1 层看第 2 层）

**第 4 步的"反向测试"是精妙**——验证"如果第 1 层被绕过，第 2 层能捕获"。

### 3.8 "现实会话示例" 的具体性

```
Bug：空的 projectDir 导致在源代码中进行 git init

数据流：
1. 测试设置 → 空字符串
2. Project.create(name, '')
3. WorkspaceManager.createWorkspace('')
4. git init 在 process.cwd() 中运行

添加了四层：
- 第 1 层：Project.create() 验证非空/存在/可写
- 第 2 层：WorkspaceManager 验证 projectDir 非空
- 第 3 层：WorktreeManager 拒绝在测试中临时目录外进行 git init
- 第 4 层：在 git init 之前记录堆栈跟踪

结果：所有 1847 个测试通过，bug 不可能重现
```

**4 步数据流 + 4 层防御 + 结果量化**——让 LLM 看到真实会话中的多层防御如何工作。

**"1847 个测试通过，bug 不可能重现"** 是关键成果——量化多层防御的价值。

### 3.9 "关键洞察" 的反教条

> 所有四层都是必要的。在测试期间，每层都捕获了其他层遗漏的 bug：
> - 不同的代码路径绕过了入口验证
> - Mocks 绕过了业务逻辑检查
> - 不同平台上的边缘情况需要环境守卫
> - 调试日志识别结构性的误用
>
> **不要停在单一验证点。在每层都添加检查。**

**4 句反教条**精准打击 LLM 倾向"加 1 个验证就够了" 的失败模式：

- 单一代码路径 → 入口验证捕获
- 不同代码路径 → 入口验证绕过 → 业务验证捕获
- mocks → 业务验证绕过 → 环境守卫捕获
- 平台特定 → 环境守卫不够 → 调试日志捕获

**每一层都被某种特定方式绕过过**——这就是为什么需要多层。

## 六、缺点 / 风险

### 6.1 4 层可能"过度防御"

在简单场景中，4 层可能过度：

- 单个函数的小工具 → 1 层入口足够
- 关键安全操作 → 4 层必要
- 关键业务逻辑 → 2-3 层合适

可以补充"什么场景需要几层" 的指导。

### 6.2 缺乏"层与层之间如何协同"的指导

4 层各自独立，但实际中：

- 入口验证失败 → 是否需要业务验证？
- 业务验证失败 → 是否需要环境守卫？
- 错误消息应当一致吗？

可以补充"层间协同" 的指导。

### 6.3 缺乏"性能影响" 的评估

多层验证增加运行时开销：

- 4 层验证 → 4 次检查
- 在 hot path 中可能影响性能
- LLM 可能担心"4 层太慢"

可以补充"性能 vs 安全" 的平衡。

### 6.4 缺乏"维护成本" 的评估

4 层 = 4 处代码修改：

- 一处数据格式变更 → 4 处都要改
- LLM 可能嫌麻烦 → 不加多层

可以补充"维护成本 vs 防御价值" 的平衡。

### 6.5 "调试日志" 的存储成本

第 4 层的 `logger.debug` 在生产中可能：

- 写入大量日志
- 占用磁盘
- 影响性能

可以补充"调试日志的开关" 策略。

### 6.6 "环境守卫" 的环境判断可能错

`process.env.NODE_ENV === 'test'` 假设所有测试都设了这个 env var：

- 没设 → 守卫失效
- 错设 → 守卫误触
- 不同测试框架设置不同

可以补充"环境判断的健壮性" 指导。
