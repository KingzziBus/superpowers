## 一、文档目的

这个技能定义了"AI 拿到一份计划后**应该怎么做**"的最小可行协议。

具体目标：
1. **承接 writing-plans 技能的输出**：本技能是 `writing-plans` 的下游消费者
2. **防止盲目执行**：强制"先批判性审阅，后执行"
3. **触发 finishing-a-development-branch**：把开发工作闭环

定位上属于 **"工作流中间环节"**——既不是起点（写计划），也不是终点（合并/清理），而是"中间的执行者"。

## 二、核心技术决策

### 决策 1：明确的"宣告协议"

```text
"I'm using the executing-plans skill to implement this plan."
```

**为什么强制宣告**：
- 让用户**知道当前处于哪个工作流阶段**
- 防止"AI 默默开始干活"的失控感
- 利于回溯——如果出问题，能定位"从哪一步开始出错"

这是 SKILL 体系的**通用约定**（其他技能也用同样模式）。

### 决策 2：上游/下游技能的显式声明

```text
上游: writing-plans → executing-plans → finishing-a-development-branch
```

**为什么**：
- 让技能"知道自己不是孤岛"——它只是工作流的一环
- 提示 AI 在执行前/后要正确链接
- 减少"AI 不知道该用哪个技能"的决策疲劳

### 决策 3：建议"有子代理就用 subagent-driven-development"

```text
如果平台支持子代理（如 Claude Code / Codex），用 superpowers:subagent-driven-development
而不是本技能。
```

**设计意图**：
- 明确告知"本技能是次优选择"
- 当条件具备时，**降级为备胎**
- 这是一种诚实的工程态度——不说自己"最好"

**潜在矛盾**：为什么要保留"次优"技能？答：
- 不是所有平台都支持子代理（如某些编辑器集成）
- 子代理消耗更多 token
- 对于简单任务，单代理执行更高效

### 决策 4：硬性 STOP 条件

```text
STOP executing immediately when:
- Hit a blocker
- Plan has critical gaps
- You don't understand an instruction
- Verification fails repeatedly
```

**关键洞察**：把"停下来"和"问问题"作为**硬性规则**而非"建议"。

**为什么重要**：
- LLM 有"强行推进"的倾向（cost pressure、time pressure）
- 把"停止"列为命令而非选项，减少 AI"找借口硬闯"的概率

### 决策 5：批判性审阅先于执行

```text
Step 1: 读计划
Step 2: 批判性审阅 (识别任何疑问)
Step 3: 如果有顾虑 → 提出；如果没有 → 继续
```

**设计意图**：
- 把"质疑"前置，而不是"先做，做完再说"
- 防止"做了一堆发现计划错了"的浪费

**与"快速执行"的张力**：批判性审阅会拖慢速度。这是 trade-off——文档默认"质量 > 速度"。

### 决策 6：永远不要未经同意在 main/master 上动手

```text
Never start implementation on main/master branch without explicit user consent
```

**为什么单独列出**：
- main/master 是"生产分支"，破坏成本极高
- LLM 默认假设"可以开始工作"——这条规则是反假设
- 与 `using-git-worktrees` 技能呼应——强制隔离工作空间

### 决策 7：流程循环设计

```
Step 1 (审阅) → 反馈 → 回到 Step 1
```

**为什么设计循环**：
- 计划可能根据反馈调整，需要重新审阅
- 避免"改完计划后硬闯"的错误

## 四、优点

### 1. 极简
- 71 行把"如何执行计划"讲清楚
- 不啰嗦、不堆砌示例
- 每个章节都可以在 30 秒内读完

### 2. 明确的工作流定位
- 知道自己在"写作-执行-完成"三阶段中的中间位置
- 不试图"重新定义"端到端流程
- 与相邻技能（writing-plans, finishing-a-development-branch）形成"职责链"

### 3. 诚实的"次优"标签
- 主动声明"有子代理就用 subagent-driven-development"
- 不掩盖本技能的局限性
- 让用户能根据上下文做最优选择

### 4. 反 LLM 冲动的硬性规则
- 4 条 STOP 条件都是"禁止强行推进"
- 强制"问问题而不是猜测"
- 减少 LLM 常见的"硬闯死循环"问题

### 5. 集成关系清晰
- 明确列出 3 个必需的工作流技能
- 上下游关系一目了然

## 五、缺点与风险

### 1. 缺少"计划不完整时如何补充"的指导

**问题**：如果计划有 gap，AI 应该：
- 是停下来问？
- 还是自己补全再继续？
- 还是先做能做的，把 gap 标出来？

**文档只说"STOP"，没说"STOP 之后具体怎么办"**。

**改进方向**：增加"发现 gap 时如何汇报"模板。

### 2. 缺少"如何处理多分支计划"的工作流

**场景**：计划涉及多个 git 分支（先做 A，再切分支做 B）。

**文档缺失**：完全没提分支切换的指导。

**与 using-git-worktrees 技能的关系模糊**——本技能说"在 main/master 之前要 consent"，但没说"多分支计划的执行顺序"。

### 3. "批判性审阅"标准不明确

**问题**：
- 怎么算"审阅通过"？
- 审阅通过后可以开始，还是审阅到多少深度才够？

**缺失**：审阅深度的客观标准。

### 4. 与 subagent-driven-development 的边界模糊

**问题**：
- subagent-driven-development 强调"派子代理"，本技能强调"自己做"
- 什么情况下该用哪个？
- 文档说"有子代理就用 subagent"——但子代理什么时候才"够用"？

**边界缺失**：没有具体的判断标准（如"任务数 < 3 用本技能"）。

### 5. 缺少进度报告协议

**问题**：
- 计划执行 30 分钟没动静，AI 怎么向用户报告？
- 阶段性完成（5/10 任务）后，是否要主动通知？

**缺失**：进度同步的节奏。

### 6. 没有"中途变更计划"的处理

**场景**：执行到第 5 步发现需要新步骤。

**文档缺失**：没说"加步骤"是停下来问，还是直接插入。

### 7. 缺少跨会话连续性

**场景**：用户说"上次我们执行到第 3 步，继续"——AI 怎么定位进度？

**缺失**：恢复会话时的状态识别。

## 六、关键洞察

### 洞察 1：技能是"工作流的中间环节"

```
┌─────────────────┐     ┌─────────────────┐     ┌────────────────────┐
│ writing-plans   │ ──→ │ executing-plans │ ──→ │ finishing-a-dev-   │
│ (写计划)         │     │ (执行计划)       │     │ branch (完成分支)   │
└─────────────────┘     └─────────────────┘     └────────────────────┘
```

**本技能只是中间一环**——它不负责"开始"或"结束"，只负责"执行"。

**启示**：好的技能设计是**单一职责 + 明确上下游**。

### 洞察 2：硬性 STOP 规则是"反 LLM 冲动的防御工事"

LLM 容易"为了完成任务而硬闯"——这是训练数据里的 bias。

```text
硬性规则：
- 缺依赖？STOP
- 测试失败？STOP
- 不理解？STOP
- 反复验证失败？STOP
```

**设计意图**：把"停下来"变成**命令式**而非"建议式"。

### 洞察 3：诚实标记自己是"次优"

```text
如果有子代理就用 subagent-driven-development，而不是本技能
```

**反工程思维**：大部分工具都试图让自己"看起来最强大"，但本技能主动说"用别的更好"。

**为什么这样做**：
- 维护者意识到**上下文决定最优**——简单任务用本技能，复杂任务用 subagent
- 不让用户陷入"只能选 executing-plans"的困境

### 洞察 4："批判性审阅"是"质量门"

```text
Step 1: Read plan
Step 2: Review critically
Step 3: If concerns → raise; else → proceed
```

**类比**：软件工程中的"代码审查"——在动手前先 review。

**目的**：把"质疑"前置，避免"做完发现方向错"的浪费。

### 洞察 5：3 步流程的"反敏捷"立场

```text
读 → 审 → 做（3 步，不可压缩）
```

**对比敏捷**："小步快跑、迭代试错"。

**本技能立场**：计划已经写好就**不要"边做边改"**——先审一遍再动手。

**适用场景**：**计划驱动**的开发，不是"探索式编程"。

## 九、给读者的启示

### 如果你刚拿到一份计划
- **宣告**："I'm using the executing-plans skill"
- **先读** + **先审**（不要上来就做）
- **遇到 STOP 条件就停**——不要硬闯
- **完成后**切到 finishing-a-development-branch

### 如果你在写计划（被本技能消费）
- **写清楚每个步骤**（不要"实现 X 功能"这种粗粒度）
- **明确验证标准**（"测试通过"要有具体命令）
- **标好上下游**（本技能会引用其他技能）

### 如果你在使用 Superpowers 体系
- **本技能是最轻的 SKILL 之一**——它的价值在于"知道什么时候不用自己"
- **如果你的平台支持子代理**，优先用 subagent-driven-development

### 如果你在做 AI 工作流编排
- **借鉴**：
  - "硬性 STOP 条件"作为反 LLM 冲动的防御
  - "诚实标记次优"作为工具链的礼貌
  - "单一职责 + 明确上下游"作为工作流设计原则

### 如果你是 LLM 训练者
- 本技能是**"少做"的样本**——明确告诉 LLM"什么时候不要做什么"
- 训练 LLM "识别 STOP 条件" 比 "完成更多任务" 更重要
