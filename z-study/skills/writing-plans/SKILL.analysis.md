## 一、文档目的

解决 Superpowers 体系中**最关键的输入**问题：**如何写出 AI 和人类都能照做的实施计划**。

具体目标：
1. **建立假设**：工程师"零上下文、品味可疑"
2. **任务粒度**：每个步骤 2-5 分钟
3. **TDD 内化**：每个任务内置 red-green-refactor
4. **反占位符**：明确禁止 "TBD"、"TODO" 等
5. **执行交接**：让用户选 subagent-driven 或 inline execution

定位上属于 **"实施计划的形式化"**——把"写计划"从艺术变成可重复的工程。

## 二、核心技术决策

### 决策 1：工程师假设是"零上下文 + 品味可疑"

```text
编写综合性的实施计划，假设工程师对我们代码库零上下文且品味可疑。
```

**设计意图**：
- 不假设工程师"应该懂"——**写下来**
- 不假设工程师"品味好"——**给出 DRY/YAGNI 指导**
- 一切**显式说明**——不留给"常识"

**这是"防御性文档"**——为最差情况写。

### 决策 2：保存位置标准化

```text
docs/superpowers/plans/YYYY-MM-DD-<feature-name>.md
```

**优点**：
- 日期前缀 = 时间序
- 目录集中 = 易于查找
- 命名一致 = 易于脚本处理

### 决策 3：范围检查前置

```text
如果规范涵盖多个独立子系统... 建议分解为多个单独的计划
```

**关键洞察**：
- 一个大计划 → 风险高（任何环节错都崩）
- 多个小计划 → 风险分散（一个坏不影响其他）
- **早分解 > 晚分解**

**这是"分而治之"原则**。

### 决策 4：文件结构先于任务

```text
在定义任务之前，绘制将要创建或修改的文件以及每个文件的职责。
```

**设计意图**：
- 先看**整体结构**——分解决策
- 再看**任务序列**——执行细节
- **架构 → 任务** 的顺序

**这是"自顶向下"原则**。

### 决策 5：一起改的文件放一起

```text
一起改的文件应该放在一起。按职责拆分，而不是按技术层。
```

**关键洞察**：
- 改 A 必改 B → A 和 B 放一起
- 避免"分布式修改"——读者要在多个目录间切换
- **耦合度高的文件应在物理上接近**

**这是"局部性原则"（locality）**。

### 决策 6：一口大小（2-5 分钟）

```text
每个步骤是一个动作（2-5 分钟）：
- "Write the failing test" - 步骤
- "Run it to make sure it fails" - 步骤
- "Implement the minimal code to make the test pass" - 步骤
- "Run the tests and make sure they pass" - 步骤
- "Commit" - 步骤
```

**精妙之处**：
- 2-5 分钟 = **可执行单元**
- 太短 = 步骤过多 → 噪音
- 太长 = 难以调试 → 不知道哪步错

**这是"原子化"原则**。

### 决策 7：TDD 内化

```text
步骤 1: 写失败的测试
步骤 2: 跑测试确认失败
步骤 3: 写实现
步骤 4: 跑测试确认通过
步骤 5: 提交
```

**精妙之处**：
- TDD 不只是"建议"——是**任务结构**
- 每 5 步 = 一个完整的 red-green-refactor 循环
- **TDD 作为任务骨架**

**这是"流程内置化"**。

### 决策 8：禁止占位符的"完整反例列表"

```text
"TBD"、"TODO"、"implement later"、"fill in details"
"Add appropriate error handling" / "add validation" / "handle edge cases"
"Write tests for the above"（不带实际测试代码）
"Similar to Task N"
描述"做什么"但不展示"怎么做"的步骤
引用未定义的类型、函数或方法
```

**关键洞察**：
- 列出**6 类**典型占位符反模式
- 每一类都是 LLM **实际会写的** 模式
- **显式禁止** = 反 LLM 偷懒

**这是"反 LLM 占位符偏好"的硬规则**。

### 决策 9：3 项自审清单

```text
1. 规范覆盖：每个需求都有任务？
2. 占位符扫描：还有 TBD/TODO 吗？
3. 类型一致性：clearLayers() vs clearFullLayers() 一致吗？
```

**关键洞察**：
- 自审是**写完计划后**必做
- 三项检查覆盖**最常见的失败模式**
- **不是分派子代理**——是 AI 自己审

**这是"质量门"**。

### 决策 10：执行交接的 2 个选项

```text
Subagent-Driven (recommended) → superpowers:subagent-driven-development
Inline Execution → superpowers:executing-plans
```

**精妙之处**：
- **不强制**一个执行方式——让用户选
- 明确**推荐** subagent-driven（更适合复杂任务）
- 显式说**Inline 是次优**——诚实

**这是"诚实标记次优"**。

### 决策 11：DRY / YAGNI / TDD 三大原则

```text
DRY — Don't Repeat Yourself
YAGNI — You Aren't Gonna Need It
TDD — Test-Driven Development
```

**关键洞察**：
- 三个**核心原则**反复出现
- 不发明新原则——用社区共识
- **简单、易记、广泛接受**

## 四、优点

### 1. 工程师假设"零上下文 + 品味可疑"

**优点**：
- 不依赖"工程师应该懂"
- 一切显式说明
- 防御性文档

### 2. 任务粒度 2-5 分钟

**优点**：
- 可执行单元
- 不会太细或太粗
- 易于调试

### 3. TDD 内化为任务结构

**优点**：
- TDD 不是"建议"——是骨架
- 每 5 步 = 一次 red-green
- **纪律性**通过结构强制

### 4. 6 类占位符反例

**优点**：
- 直接戳穿 LLM 偷懒模式
- 显式禁止
- 防止"看起来完整但实际空洞"的计划

### 5. 3 项自审清单

**优点**：
- 写完必做
- 覆盖最常见失败
- AI 自己审（不依赖子代理）

### 6. 2 个执行选项

**优点**：
- 用户主权
- 显式推荐 subagent-driven
- 诚实标记 Inline 是次优

### 7. DRY/YAGNI/TDD 三大原则

**优点**：
- 用社区共识
- 易记、广泛接受
- 不发明新原则

## 五、缺点与风险

### 1. 计划头部没列 "Risks / Open Questions"

**缺失**：
- 计划阶段的已知风险
- 跨任务依赖
- 需要用户决策的开放问题

**改进**：增加 "Open Questions" 章节。

### 2. 任务粒度 2-5 分钟可能过严

**问题**：
- 复杂任务（设计、调试）很难 5 分钟完成
- 严格 2-5 分钟 = 拆分过度

**缺失**：粒度的灵活性。

### 3. TDD 不适用所有任务

**问题**：
- 文档任务没"测试"
- 配置任务没"测试"
- UI 任务难 TDD

**缺失**：非 TDD 任务的替代流程。

### 4. 工程师假设"品味可疑"但没给"具体标准"

**问题**：
- "品味"太主观
- 应该给具体的代码风格、命名约定

**缺失**：具体的代码标准。

### 5. 缺"计划评审"机制

**问题**：
- 写完计划后，AI 自审
- 但**没让用户审**——用户只能在执行时发现错误

**改进**：用 `plan-document-reviewer-prompt.md`（本目录另一个文件）做用户层审查。

### 6. "频繁提交"没量化

**问题**：
- 每次任务后提交 ✓
- 但**大任务**应该多次提交

**缺失**：提交频率的具体标准。

### 7. 缺"计划维护"指南

**问题**：
- 计划写完后，执行中会偏离
- 偏离后怎么更新计划？

**缺失**：计划维护协议。

### 8. "文件结构" 章节没示例

**问题**：
- "按职责拆分，不是按技术层"——但**没说怎么做**
- 抽象原则缺具体例子

**改进**：增加"文件结构"示例图。

## 六、关键洞察

### 洞察 1：这是"计划的形式化"——把艺术变工程

**观察**：
- 传统"写计划"靠经验——不同人写得不一样
- LLM 写计划 = 容易空洞（占位符、抽象）

**本技能的价值**：
- 6 类占位符 = **反 LLM 偷懒**
- TDD 任务结构 = **纪律性内置**
- 2-5 分钟粒度 = **可执行单元**
- 3 项自审 = **质量门**

**这是"AI 时代计划工程"**。

### 洞察 2：工程师假设"零上下文 + 品味可疑"是"防御性文档"

**关键洞察**：
- 计划服务于"未来读它的人"——可能是别的开发者
- **"零上下文"**假设 = 一切显式
- **"品味可疑"**假设 = 显式标准

**这是"反人类记忆偏好"的文档**。

### 洞察 3：TDD 作为任务骨架是"流程内置化"

```text
写失败测试 → 跑测试 → 写实现 → 跑测试 → 提交
```

**关键洞察**：
- TDD 不只是"建议用"——是**任务结构本身**
- 每个任务 5 步 = 一次 red-green
- **纪律通过结构强制**——不靠意志

**这是"纪律结构化"原则**。

### 洞察 4：6 类占位符是"反 LLM 偷懒模式"

**观察**：
- LLM 训练数据里**充斥着占位符**——"TBD"、"TODO"、"implement later"
- 这些看起来"完整"——实际**没内容**

**本技能的价值**：
- 显式列出 6 类**LLM 最常写的占位符**
- **每类都禁止**——不留模糊

**这是"反 LLM 训练偏差"的硬规则**。

### 洞察 5：3 项自审是"质量门"

```text
1. 规范覆盖
2. 占位符扫描
3. 类型一致性
```

**关键洞察**：
- 写完计划后**必做**——不是"可选优化"
- 三项检查**覆盖最常见失败**
- **AI 自己审**（不依赖子代理）——降低协调成本

**这是"质量门的形式化"**。

### 洞察 6：2 个执行选项是"诚实标记次优"

```text
Subagent-Driven (recommended)
Inline Execution
```

**关键洞察**：
- 不强制——**用户主权**
- 显式推荐 subagent-driven——承认优势
- 显式说 Inline 是次优——**诚实**

**这是"用户主权 + 诚实"**。

### 洞察 7：DRY/YAGNI/TDD 是"借用社区共识"

**关键洞察**：
- 不发明新原则——用社区广泛接受的原则
- **简单、易记、有大量工具支持**
- 让"计划原则"和"工程实践"对齐

**这是"站在巨人肩膀上"**。

### 洞察 8：与 executing-plans / subagent-driven-development 的三角

```
writing-plans (上游)
   ↓
executing-plans (inline) ─ OR ─ subagent-driven-development (recommended)
   ↓
finishing-a-development-branch (收尾)
```

**关键洞察**：本技能是**起点**——决定了后续执行方式。

## 九、给读者的启示

### 如果你是 AI 代理
- **每个计划** 必含头部（Goal / Architecture / Tech Stack）
- **每个任务** 必含完整代码（不能"TBD"）
- **每个步骤** 2-5 分钟
- **每个任务** TDD 内化
- **写完** 必做 3 项自审

### 如果你是 LLM 训练者
- **本技能是"反 LLM 偷懒"的核心训练数据**
- 6 类占位符 = **必须禁止的训练偏差**
- 任务结构 = **TDD 内化的训练样本**

### 如果你是 AI 工具开发者

  - "工程师假设零上下文"作为**默认文档标准**
  - "TDD 内化"作为**任务结构**
  - "6 类占位符"作为**生成器的反例集**
  - "3 项自审"作为**质量门**
