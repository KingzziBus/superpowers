## 2. 目的与背景

### 2.1 文档的双重身份

这份文档是 `writing-skills/SKILL.md` 中"反合理化"章节的**理论支撑**：
- 文档本身说"See persuasion-principles.md for research foundation"
- 它解释了为什么 "YOU MUST" 比 "Consider" 有效
- 它提供了按技能类型选择说服策略的指南

### 2.2 核心命题

> "LLMs respond to the same persuasion principles as humans. Understanding this psychology helps you design more effective skills - not to manipulate, but to ensure critical practices are followed even under pressure."

**关键限定："not to manipulate"** —— 说服技术的使用必须服务用户的真正利益。

### 2.3 解决的三个问题

| 问题 | 本文档提供的方案 |
|------|------------------|
| 文档被 agent 忽略 | 权威 + 承诺 + 社会认同的组合 |
| 文档被 agent 钻空子 | "YOU MUST" + "No exceptions" 的命令式语言 |
| 文档被 agent 误用 | 合乎道德的使用边界 |

## 3. 核心技术决策

### 3.1 决策 1：应用 Cialdini 七大原则而非自创框架

**为什么引用 Cialdini？**

| 优势 | 论证 |
|------|------|
| 学术权威 | Cialdini 是说服研究的奠基人 |
| 实证基础 | 7 原则有 50+ 年研究支持 |
| 易于理解 | 公众已熟悉的框架 |
| 跨文化适用 | 七原则在多文化中被验证 |

**七大原则的层次：**

```
高频使用（推荐）：
├── 权威（Authority）
├── 承诺（Commitment）
├── 稀缺性（Scarcity）
├── 社会认同（Social Proof）
└── 统一（Unity）

低频/避免：
├── 互惠（Reciprocity）—— 谨慎使用
└── 喜爱（Liking）—— 总是避免（纪律类）
```

### 3.2 决策 2：明确标记"不要用"的两个原则

> "Reciprocity: Use sparingly - can feel manipulative"
> "Liking: DON'T USE for compliance - Creates sycophancy"

**这一决策的重要性：**
- 互惠的"恩惠"模式在 AI 上下文中容易被视为操纵
- 喜爱原则与诚实的反馈文化冲突
- 文档明确说"会创造谄媚"（sycophancy）—— 这正是 LLM 已知问题

### 3.3 决策 3：按技能类型映射原则组合

| 技能类型 | 使用原则 | 避免原则 | 理由 |
|----------|----------|----------|------|
| 纪律执行 | 权威 + 承诺 + 社会认同 | 喜爱、互惠 | 需要绝对性 |
| 指导/技术 | 中等权威 + 统一 | 重度权威 | 需要灵活性 |
| 协作 | 统一 + 承诺 | 权威、喜爱 | 需要平等性 |
| 参考 | 仅清晰度 | 所有说服 | 需要客观性 |

**这一映射的关键洞察：**
- 不是所有技能都需要"重度说服"
- 参考类技能（API 文档）说服反而会降低质量
- 协作类技能需要"我们"语言而非"你必须"

### 3.4 决策 4：每个原则的"它如何起作用"具体化

**不只是描述原则，而是给出具体技术：**

**权威：**
- 命令式语言："YOU MUST"、"Never"、"Always"
- 不可协商的框架："No exceptions"
- 消除决策疲劳

**承诺：**
- 要求声明："Announce skill usage"
- 强制明确选择："Choose A, B, or C"
- 使用 TodoWrite 跟踪

**稀缺性：**
- 时间限制要求："Before proceeding"
- 顺序依赖："Immediately after X"
- 防止拖延

**社会认同：**
- 通用模式："Every time"、"Always"
- 失败模式："X without Y = failure"
- 建立规范

**统一：**
- 协作语言："our codebase"、"we're colleagues"
- 共享目标："we both want quality"

**这种"原则→技术→示例"的三段式是 LLM-friendly 的——** 文档不仅告诉 LLM "为什么"，还告诉它"怎么做"。

### 3.5 决策 5：实证数据驱动（Meincke 2025）

**关键数据：**
- N=28,000 LLM 对话
- 遵从率：33%（无说服）→ 72%（有说服）
- p < .001（统计显著）
- 最有效的原则：权威、承诺、稀缺性

**价值：** 不是"哲学思辨"，是"实证数据"。这让 superpowers 的说服技术有学术合法性。

### 3.6 决策 6：合乎道德的边界明确化

**合法：**
- 确保关键实践被遵循
- 创建有效的文档
- 防止可预测的失败

**不合法：**
- 为个人利益操纵
- 制造虚假的紧迫感
- 基于内疚的遵从

**道德测试：**
> "Would this technique serve the user's genuine interests if they fully understood it?"

**这一决策的意义：**
- 把"操纵"与"说服"区分开
- 强调"用户真正利益"而非"agent 表面遵从"
- 给"什么时候停止"提供标准

### 3.7 决策 7："LLM 是类人的"假设

> "LLMs are parahuman"

**含义：**
- LLM 在人类文本上训练，包含说服模式
- 权威语言在训练数据中"先于"遵从
- 承诺序列（声明→行动）被广泛建模
- 社会认同模式（每个人都做 X）建立规范

**意义：** 这是一个"机制性解释"——为什么说服技术对 LLM 有效，不是因为 LLM "理解" 说服，而是因为训练数据中的统计规律。

## 5. 优点 / 亮点

### 5.1 范式创新：把说服心理学武器化

传统上，说服心理学用于：
- 销售、营销
- 政治传播
- 谈判

本文档的创新：**把它用于工程文档**——技能设计。这是一种"技术化说服"。

### 5.2 实证数据驱动

> "Tested 7 persuasion principles with N=28,000 AI conversations. Persuasion techniques more than doubled compliance rates (33% → 72%, p < .001)"

这不是"经验之谈"，是"实验数据"。让 superpowers 的说服技术有学术合法性。

### 5.3 "不操纵"的道德边界

> "not to manipulate, but to ensure critical practices are followed"

明确的道德立场——说服是为了**用户利益**，不是为了让 agent 表面遵从。

### 5.4 双向原则清单（用 + 不用）

不只说"用什么原则"，还说"避免什么原则"。**这种"反例"是高质量技术文档的标志**。

### 5.5 "原则→技术→示例"三段式

每个原则都有：
- "它是什么"（What）
- "它如何起作用"（How）
- "何时使用"（When）
- "示例"（Example）

这种结构让 LLM 能**直接应用**而不只是"理解"。

### 5.6 按技能类型的原则映射

4 类型 × 原则矩阵是**实践指南**——把抽象原则转化为可决策的表格。

### 5.7 "执行意图"的引用

> "Implementation intentions create automatic behavior"
> "Clear triggers + required actions = automatic execution"

引用心理学中的"执行意图"概念，为"When X, do Y"模式提供学术支撑。

### 5.8 "类人"假设的诚实

> "LLMs are parahuman"

不假装 LLM "理解"说服，而是说它们"响应"说服模式。这种诚实是科学态度。

### 5.9 双向引用（Cialdini + Meincke）

Cialdini 提供经典理论，Meincke 提供 LLM 特异性数据。**两者结合**让文档既有深度又有针对性。

### 5.10 "快速参考"的实用性

5 个设计问题是**可操作的清单**——技能设计者可以自我审视。

## 6. 缺点 / 风险 / 局限

### 6.1 道德边界的"主观性"风险

> "Would this technique serve the user's genuine interests if they fully understood it?"

**问题：**
- "真正利益"由谁定义？
- 不同利益相关者可能有不同"真正利益"
- 文档没有给出客观判断标准

**风险：** "用户利益"可能被工具设计者重新定义。

### 6.2 "互惠"和"喜爱"原则的过度简化

> "Reciprocity: Use sparingly - can feel manipulative"
> "Liking: DON'T USE for compliance"

**问题：**
- "互惠"在某些场景下可能是合理的（如"我帮你 → 你帮我"）
- "喜爱"完全避免可能让协作技能显得冷漠
- 文档没有讨论"灰色地带"

### 6.3 缺少量化的"原则强度"

文档说权威、承诺、稀缺性"最有效"，但：
- 三者的相对强度？
- 一个技能应使用多少"剂量"的原则？
- 过度使用某原则的副作用？

**风险：** agent 可能"过度说服"——把所有原则都用上反而让技能显得操纵。

### 6.4 "类人"假设的局限性

> "LLMs are parahuman"

**问题：**
- 不同 LLM 的"类人"程度不同
- 多模态模型（如 GPT-4V）的"响应模式"可能不同
- 训练数据中的偏见会传递到"说服响应"

**风险：** 把"训练数据中的统计规律"当作"普遍真理"。

### 6.5 缺少"跨文化"考量

Cialdini 的原则主要基于西方研究。Meincke 的实验可能也是西方中心的。

**问题：**
- 集体主义文化可能对"权威"和"统一"响应不同
- 个人主义文化可能对"承诺"和"稀缺性"响应不同
- 文档没有讨论跨文化适用性

### 6.6 "合乎道德"边界的灰色地带

**举例：**
- "稀缺性"是"防止拖延"还是"制造虚假紧迫感"？
- "权威"是"提供可靠来源"还是"压迫异议"？
- "承诺"是"建立问责"还是"剥夺灵活性"？

**风险：** 同一原则可能被解释为合法或非法。

### 6.7 与 writing-skills 的"过度依赖"风险

如果技能设计者过度依赖说服技术，可能：
- 替代了"技能本身的质量改进"
- 制造"看似防弹但实际糟糕"的技能
- 文档没有讨论"技能质量" vs "说服强度" 的权衡

### 6.8 缺少失败模式分析

文档没说"什么原则在什么场景下会失败"：
- 权威的失败：agent 可能"反权威"（不遵从专家）
- 承诺的失败：agent 可能"过度承诺"（承诺超出能力）
- 稀缺性的失败：agent 可能"焦虑遵从"（违反技能以求快速完成）

### 6.9 "类比"的过度推论

文档用"机器人"、"桥梁"等类比解释概念，但：
- 类比是有限度的
- 不同文化对类比的接受度不同
- 文档没有声明"类比的边界"

### 6.10 实验数据的可重复性

Meincke et al. (2025) 的研究：
- N=28,000 是大样本
- 但具体场景、模型、提示词是什么？
- 复现性如何？

**风险：** 文档引用了实验但没有详细方法论。

## 7. 关键洞察

### 7.1 洞察 1：说服技术是 LLM 文档学的"隐藏维度"

传统文档学关注：准确性、完整性、可读性、教育性。

**本文件新增维度：说服性** —— 让文档不仅"正确"还要"被遵从"。

这是 LLM 时代文档学的**范式转变**。

### 7.2 洞察 2：权威的"绝对性"消除合理化空间

> "'YOU MUST' removes decision fatigue"
> "Absolute language eliminates 'is this an exception?' questions"

**机制：** 绝对语言关闭"例外思考"——agent 不再问"这是不是特殊情况"。

这是抗合理化的**核心技术**。

### 7.3 洞察 3：执行意图的"自动化"效应

> "Clear triggers + required actions = automatic execution"
> "'When X, do Y' more effective than 'generally do Y'"

**机制：** "When X, do Y" 创建**情境-行动**绑定，让遵从变成"反射"而非"决策"。

这是把"判断"转化为"模式匹配"。

### 7.4 洞察 4：社会认同的"统计常态化"

> "Universal patterns: 'Every time', 'Always'"
> "X without Y = failure"

**机制：** 把某种行为"常态化"——"每个人都做"成为 agent 的默认假设。

**风险：** 可能制造"群体思维"——即使某个行为是错的，如果被描述为"每个人做"，agent 也会遵从。

### 7.5 洞察 5：互惠和喜爱的"剥削性"是 LLM 已知弱点

LLM 已知的"谄媚"（sycophancy）问题——过度赞同用户——本质上是"喜爱"原则的副产品。

文档明确说**"DON'T USE"** 这一原则，是对 LLM 已知问题的**预防性约束**。

### 7.6 洞察 6：道德边界是"用户利益"而非"agent 遵从"

> "Would this technique serve the user's genuine interests?"

**关键：** 文档把"成功"定义为"用户受益"，不是"agent 遵从"。

**意义：** 即使 agent 表面遵从但损害用户利益，这也是"操纵"而非"说服"。

### 7.7 洞察 7：原则组合是"配方"而非"列表"

> "Discipline-enforcing: Authority + Commitment + Social Proof"

**洞察：** 不是用单个原则，而是用原则**组合**形成"配方"。纪律类技能需要三重锁定。

### 7.8 洞察 8："类人"假设的"双刃剑"

**正面：** 可以借用成熟的说服心理学。
**负面：** LLM 也继承了人类的认知偏见（如权威崇拜、从众心理）。

**风险：** 同一技术可被用于"善"（确保安全实践）或"恶"（操纵用户）。

### 7.9 洞察 9：参考类技能不需要说服

> "Reference: Clarity only - All persuasion"

**洞察：** 说服是"行为改变工具"，不是"信息传递工具"。

**意义：** API 文档不需要"权威"——需要的是"清晰"。

### 7.10 洞察 10：设计问题是"自我审视"工具

5 个设计问题（什么类型 / 改变什么 / 哪个原则 / 太多吗 / 合乎道德吗）是**元认知工具**——让设计者反思自己的设计决策。

3. **"When X, do Y"** 创建执行意图
4. **避免"互惠"和"喜爱"**——它们会创造操纵或谄媚
5. **每个技能类型用不同的原则组合**——参考类不要说服

### 10.2 对 AI 工程师

1. **说服性是 LLM 文档的新维度**——比传统文档学多一个优化目标
2. **"类人"假设是双刃剑**——既能用也能被滥用
3. **道德边界是"用户利益"**——不是"agent 表面遵从"
4. **过度说服有副作用**——制造"看似防弹但实际糟糕"的技能
5. **Meincke 2025 是关键证据**——N=28,000 验证说服技术

### 10.3 对 superpowers 用户

1. **TDD/verification 技能的"铁律" = 权威+承诺+社会认同**
2. **红旗列表 = 社会认同**（"这是常见失败模式"）
3. **TodoWrite 跟踪 = 承诺**（让 agent 明确承诺步骤）
4. **"我正在使用 [Skill]" = 承诺**（公开声明）
5. **"我们"语言 = 统一**（"our codebase"）

### 10.4 对更广泛的提示工程

1. **说服技术是 prompt engineering 的新前沿**
2. **"执行意图"模式（When X, do Y）值得广泛应用**
3. **避免"谄媚"——LLM 已知问题需要预防性约束**
4. **道德边界是工程伦理——不只是哲学讨论**
5. **跨文化适用性是新挑战——Cialdini 主要是西方研究**

## 11. 后续工作 / 改进方向

### 11.1 应补充的内容

1. **量化的"原则强度"**——权威 vs 承诺 vs 稀缺性的相对强度
2. **"剂量"指导**——一个技能应使用多少原则
3. **失败模式分析**——什么原则在什么场景下会失败
4. **跨文化适用性**——不同文化对各原则的响应差异
5. **实验方法的详细说明**——Meincke 2025 的具体场景、模型、提示词

### 11.2 应展开的讨论

1. **"用户利益"的客观化**——如何避免工具设计者重新定义
2. **灰色地带**——互惠/喜爱原则的合理使用场景
3. **过度说服的检测**——如何识别"看似防弹但实际糟糕"的技能
4. **元层面**：技能本身可以是"说服"的对象吗？
5. **说服技术的演进**——LLM 进化后这些技术会失效吗？

### 11.3 元层面

本文档是 superpowers 的"理论支柱"之一。建议：
- 收集 superpowers 用户对"哪些原则最有效"的反馈
- 跟踪 LLM 进化对说服技术的影响
- 定期更新道德边界（随 LLM 能力演进）

### 11.4 哲学层面

最大的开放问题是：**"为善的说服"与"操纵"的客观边界是什么**？

当前的答案是"用户利益"，但这是循环定义——谁定义"利益"？

可能的解决方案：
- 多利益相关者协商
- 透明化（公开使用什么原则）
- 可逆性（用户可以撤回承诺）
- 实证检验（说服后的实际行为是否改善用户处境）
