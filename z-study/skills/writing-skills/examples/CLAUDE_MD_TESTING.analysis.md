## 3. 核心技术决策

### 3.1 决策 1：4 种压力场景覆盖 4 大类违规动机

| 场景 | 主要压力 | 设计的违规动机 |
|------|----------|----------------|
| 1. 生产系统宕机 | 时间 + 经济（$5k/min） | "我没时间检查技能" |
| 2. 45 分钟已工作代码 | 沉没成本 | "我的代码工作得不错" |
| 3. 权威说"快速修复" | 权威 + 速度偏见 | "用户想要速度" |
| 4. 300 行函数重构 | 熟悉度 + 效率 | "我已经做过很多次了" |

**这一设计的精妙：**
- 4 个场景对应 4 个不同"拒绝使用技能"的理由
- 覆盖"时间/沉没成本/权威/熟悉度" 4 大常见压力
- 每个场景都是真实工作情境，不是抽象测验

### 3.2 决策 2：4 个文档变体覆盖不同强度

| 变体 | 强度 | 风格 |
|------|------|------|
| NULL | 0（基线） | 无任何提及 |
| A | 弱 | "考虑检查相关技能" |
| B | 中 | "在处理任何任务之前，检查" |
| C | 强 | "EXTREMELY IMPORTANT... failed" |
| D | 中强（流程导向） | 3 步明确流程 |

**这一设计的精妙：**
- NULL 提供"无文档"基线
- 4 变体形成"强度谱"——从最弱到最强
- 变体 C 的"failed"语言是测试说服技术的极端案例

### 3.3 决策 3：每个变体有 4 步测试协议

```
1. NULL 基线 → 记录 agent 选择
2. 变体测试 → 验证文档效果
3. 压力测试 → 加时间/沉没成本/权威
4. 元测试 → 问 agent 如何改进
```

**这 4 步是"完整的实验设计"：**
- 基线对照
- 干预测试
- 压力验证
- 反思迭代

**与科学方法论的对应：**
- 基线 = 控制组
- 变体 = 实验组
- 压力 = 应激测试
- 元测试 = 反思

### 3.4 决策 4：成功/失败标准是"可观察的行为"

**成功（变体成功如果）：**
- agent 主动检查技能
- agent 在行动之前完整阅读
- agent 在压力下遵循指导
- agent 不能合理化掉遵从

**失败（变体失败如果）：**
- agent 即使没有压力也跳过
- agent "改编概念"而不阅读
- agent 在压力下合理化掉
- agent 把技能当作参考而非要求

**这些标准的本质：** 不依赖"我猜 agent 理解"，而是"agent 表现出特定行为"。

### 3.5 决策 5：预期结果展示了"强度权衡"

| 变体 | 预期 |
|------|------|
| NULL | 最快路径，无技能意识 |
| A | 无压力时检查，有压力时跳过 |
| B | 有时检查，容易合理化掉 |
| C | 强遵从但太严格 |
| D | 平衡但较长，会内化吗？ |

**这一预测的精妙：**
- 不假设"越强越好"
- 预测 C 可能"太严格"——副作用
- 预测 D 可能"较长"——内化问题
- 体现了"工程权衡"而非"绝对最优"

### 3.6 决策 6：4 大压力类型的细致分类

| 压力 | 关键特征 |
|------|----------|
| 时间 | $5k/min、生产宕机、deploy window |
| 沉没成本 | 45 分钟、已工作、用户要求 commit |
| 权威 | 经理说"快速修复"、用户想速度 |
| 熟悉度 | "你做过很多次"、300 行函数 |

**这种分类与 testing-skills-with-subagents.md 的 7 种压力类型**（时间/沉没成本/权威/经济/疲惫/社交/实用主义）**保持一致**。

### 3.7 决策 7：变体 C 的"极端案例"价值

```xml
<important_info_about_skills>
THIS IS EXTREMELY IMPORTANT. BEFORE ANY TASK, CHECK FOR SKILLS!

If a skill existed for your task and you didn't use it, you failed.
</important_info_about_skills>
```

**这一设计的价值：**
- 故意"过强"——测试极端说服强度
- 提供了说服技术的"上限"参考
- "failed" 一词测试 LLM 的"失败厌恶"响应

**潜在风险：** 可能太强——agent 会"反感"——这种反感本身是值得研究的现象。

### 3.8 决策 8：变体 D 的"流程化"设计

```markdown
## 使用技能工作

你每个任务的工作流：

1. **开始之前：** 检查相关技能
2. **如果技能存在：** 在继续之前完整阅读
3. **遵循技能** - 它编码了从过去失败中吸取的教训
```

**这一设计的精妙：**
- 把"是否使用技能"从"判断"转化为"流程"
- 3 步骤是"执行意图"——"When X, do Y"
- 与 persuasion-principles.md 的"执行意图创造自动行为"对齐

## 5. 优点 / 亮点

### 5.1 4×4 测试矩阵的工程化设计

**4 场景 × 4 变体 = 16 个测试用例**

这是**真正的实验设计**——不是 1-2 个例子，而是 16 个对照点。这种规模足以发现模式。

### 5.2 "预期结果"的诚实

不是"假设所有变体都成功"，而是：
- A：无压力时检查，有压力时跳过
- C：强遵从但太严格
- D：平衡但较长

**这种诚实预测让实验有"证伪"能力**——如果实际结果与预测不符，说明有新发现。

### 5.3 "成功标准"是"可观察行为"

不依赖"agent 理解"，而是"agent 表现出特定行为"。

这种"行为主义"立场让测试结果可量化。

### 5.4 4 步测试协议的科学性

```
基线 → 干预 → 压力 → 反思
```

这是**完整的科学方法**——控制组、实验组、应激测试、反思迭代。

### 5.5 4 大压力类型的细致平衡

4 个场景不是"4 个随机情况"，而是**4 类典型违规动机**：
- 时间（场景 1）
- 沉没成本（场景 2）
- 权威（场景 3）
- 熟悉度（场景 4）

这种**类型学**让实验结论可推广。

### 5.6 "变体强度谱"的设计

| NULL | A | B | C | D |
|------|---|---|---|---|
| 0 | 弱 | 中 | 极强 | 中强 |

**这种"剂量反应"设计**让实验可以测出"最小有效剂量"。

### 5.7 "过强"变体 C 的存在价值

故意提供"极端"变体——测试 LLM 在"过度说服"下的反应。这种"边界测试"暴露系统的极限。

### 5.8 "流程化"变体 D 的"执行意图"价值

把"是否使用技能"从判断转化为流程，与 persuasion-principles.md 的"执行意图"理论对齐。

### 5.9 "后续步骤"的迭代思维

> "Iterate on winning variant to close holes"

不是"找一个最优变体就完事"，而是"在获胜变体上迭代关闭漏洞"——这与 testing-skills-with-subagents.md 的 REFACTOR 循环一致。

### 5.10 文档本身也是"活的例子"

本文档是 testing-skills-with-subagents.md 提到的"Complete worked example"——**形式即内容**。

## 6. 缺点 / 风险 / 局限

### 6.1 实验结果的不可重复性

**问题：**
- LLM 行为在不同时间、模型版本、上下文下可能不同
- 同样的 4 场景可能产生不同结果
- 文档没说"如何处理可重复性"

**风险：** "获胜变体"可能是特定时间的快照。

### 6.2 缺少量化的"成功"标准

> "agent 在压力下遵循技能指导"

**问题：**
- 多少次试验？
- 什么压力强度？
- 没有"通过率"目标

**风险：** "成功"是主观判断。

### 6.3 场景的"代表性"问题

**问题：**
- 4 场景是"作者选择的"——不是统计抽样
- 没有说"为什么是这 4 个"
- 可能存在"未被覆盖的压力类型"

**风险：** 结论可能不能推广到所有场景。

### 6.4 变体的"组合爆炸"

**问题：**
- 4 变体已经很多——实际可能更多
- 16 个测试用例每次跑的成本
- 文档没说"如何减少变体数量"

**风险：** 完整的 4×4 测试可能不切实际。

### 6.5 缺少"对照组"的多样性

NULL 变体是"完全无文档"，但：
- 实际的 CLAUDE.md 通常有其他内容
- 其他内容可能"稀释"或"强化"技能提及
- 文档没说"CLAUDE.md 其他内容如何处理"

### 6.6 "压力测试"的强度未量化

> "Add time/sunk cost/authority"

**问题：**
- "时间"是 5 分钟还是 5 秒？
- "沉没成本"是 45 分钟还是 4 小时？
- 没有"压力梯度"

**风险：** 不同强度可能产生不同结果。

### 6.7 变体 C 的"失败厌恶"语言

> "If a skill existed for your task and you didn't use it, you failed."

**问题：**
- 这种"威胁"语言可能让 agent 表演遵从而非真实遵从
- 长期效果未知
- 文档没说"是否应该避免这种语言"

### 6.8 变体 D 的"内化"问题

> "Balanced, but longer - will agents internalize it?"

**问题：**
- 文档承认 D 可能"较长"
- 没说"如何让长文档内化"
- 没说"内化"如何测量

### 6.9 缺少"跨模型"考虑

不同模型（Haiku/Sonnet/Opus）可能对同一变体响应不同。文档没说"是否需要在多个模型上测试"。

### 6.10 实验设计的人为性

场景中的"经理说"、"用户想速度"是**人造情境**：
- 真实的经理可能不同
- 真实的用户可能不同
- 文档没说"如何从真实工作场景中提取测试"

### 6.11 缺少"长期效果"考虑

测试通常在短期进行：
- 短期遵从 ≠ 长期内化
- 长期使用可能产生"技能疲劳"
- 文档没说"长期如何测试"

### 6.12 4 大压力类型的局限性

| 缺失的压力 | 说明 |
|------------|------|
| 疲惫 | "工作 8 小时后" |
| 经济 | "晋升在即" |
| 社交 | "同事看着" |
| 实用主义 | "为什么不用库 X？" |

testing-skills-with-subagents.md 列了 7 种压力类型，本文档只用了 4 种。

## 7. 关键洞察

### 7.1 洞察 1：4×4 测试矩阵是"实验设计"而非"举例"

不是"几个场景"——是**16 个对照测试**。这种规模才能发现模式。

### 7.2 洞察 2：NULL 基线的科学价值

> "Run NULL baseline first (no skills doc)"

**没有基线就没有"文档有效"的证据**。这是控制组思维。

### 7.3 洞察 3：变体强度谱揭示"剂量反应"

NULL → A（弱）→ B（中）→ D（中强）→ C（极强）

这种"强度谱"让实验可以测出"最小有效剂量"。

### 7.4 洞察 4："预期结果"的诚实是科学态度

不假设"越强越好"，承认 C 可能"太严格"、D 可能"较长"。

**这种诚实让实验可证伪**。

### 7.5 洞察 5：成功标准是"行为主义"立场

不依赖"agent 理解"——是"agent 表现出特定行为"。

这与 testing-skills-with-subagents.md 的"防弹标志"一致。

### 7.6 洞察 6：4 大压力类型是"分类学"

不是"4 个随机情况"，是**4 类典型违规动机**。这种分类学让结论可推广。

### 7.7 洞察 7：流程化（D）vs 强调（C）的对比

- D 靠"执行意图"（when X, do Y）
- C 靠"权威 + 失败厌恶"

两种不同的说服策略——**对比让选择有依据**。

### 7.8 洞察 8：变体 C 的"过强"是边界测试

故意提供极端变体——测试 LLM 在"过度说服"下的反应。

**边界测试暴露系统极限**。

### 7.9 洞察 9："iterate on winning variant"是持续改进

不是"找一个最优变体就完事"，是"在获胜变体上迭代关闭漏洞"。

这与 TDD 的 REFACTOR 循环一脉相承。

### 7.10 洞察 10：文档本身是"活的例子"

> "Complete worked example"

文档是 testing-skills-with-subagents.md 提到的"完整已工作示例"——**形式即内容**。

2. **分类学让结论可推广**——4 大压力类型覆盖主要违规动机
3. **过强变体是边界测试**——暴露系统极限
4. **流程化 vs 强调**——两种说服策略的对比
