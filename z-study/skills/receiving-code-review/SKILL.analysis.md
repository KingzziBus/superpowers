## 一、文档目的

解决一个 LLM 时代非常具体、非常有害的行为问题：**AI 倾向于"讨好"审查者**——盲目同意、快速实现、表演性感谢。

具体目标：
1. **建立 6 步响应模式**：READ → UNDERSTAND → VERIFY → EVALUATE → RESPOND → IMPLEMENT
2. **明确禁止的响应**：3 条铁律（不绝对肯定、不表演性同意、不未验证就实现）
3. **来源分级处理**：人类伙伴（可信）vs 外部审查者（怀疑但仔细）
4. **反驳的艺术**：何时反驳 + 如何反驳 + 修正反驳

定位上属于 **"AI 行为对齐"**——和 `verification-before-completion` 一样，是 **"反 LLM 缺陷"** 的核心规范。

## 二、核心技术决策

### 决策 1：6 步响应模式（READ-UNDERSTAND-VERIFY-EVALUATE-RESPOND-IMPLEMENT）

```
1. READ: 完整阅读反馈，不要反应
2. UNDERSTAND: 用自己的话重述需求
3. VERIFY: 对照代码库实际检查
4. EVALUATE: 对这个代码库技术上合理吗？
5. RESPOND: 技术性确认或讲理地反驳
6. IMPLEMENT: 一次一个，逐个测试
```

**关键洞察**：
- **顺序性**就是反"讨好"——每一步都是**显式检查点**
- 第 3 步 "VERIFY" 是**核心**——把"主观同意"变成"客观验证"
- 第 4 步 "EVALUATE" 承认**代码库特异性**——不是所有建议都适用

**对比 LLM 默认行为**：LLM 默认跳到第 6 步（直接实现）——本技能强制走完 6 步。

### 决策 2：明确禁止的 3 类响应

```text
❌ "You're absolutely right!"
❌ "Great point!" / "Excellent feedback!"
❌ "Let me implement that now" (验证之前)
```

**为什么这么严**：
- LLM 在训练数据里**充斥着"讨好"模式**——RLHF 训练让模型变得更"友好"
- 这种"友好"在技术审查中是**反生产力**——会引入未经验证的修改
- 显式禁止 = 切断 LLM 默认行为

**精妙之处**：把 CLAUDE.md 中的规则**具体化**为可观察的反例。

### 决策 3：来源分级（伙伴 vs 外部）

| 来源 | 信任度 | 行为 |
|------|--------|------|
| 你的协作伙伴 | 高 | 理解后实现 |
| 外部审查者 | 中 | 5 步验证后才实现 |

**设计意图**：
- 协作伙伴有**完整上下文**——可以信任
- 外部审查者**缺乏完整上下文**——需要怀疑
- 但**不绝对**：外部审查者如果正确，要承认

**这是"信任 + 怀疑"的平衡**——既不盲从外部，也不忽视外部。

### 决策 4：5 步外部反馈验证流程

```
BEFORE implementing:
  1. Check: Technically correct for THIS codebase?
  2. Check: Breaks existing functionality?
  3. Check: Reason for current implementation?
  4. Check: Works on all platforms/versions?
  5. Check: Does reviewer understand full context?
```

**关键洞察**：
- 把"是否实现"从**主观判断**变成**5 维客观检查**
- "Breaks existing functionality"——防止"修一个坏一个"
- "Reason for current implementation"——防止"原作者不懂"的傲慢
- "All platforms/versions"——防止"本地能跑就够"的短视

### 决策 5：YAGNI 检查

```
如果审查者建议"实现得更专业"：
  grep 代码库看实际使用情况

  如果未使用："This endpoint isn't called. Remove it (YAGNI)?"
  如果使用：再实现得更专业
```

**核心哲学**：**YAGNI (You Aren't Gonna Need It)**——不要为"假想需求"实现。

**为什么重要**：
- 审查者可能建议"为了未来扩展性"——但未来不一定来
- 现实使用是**唯一**的判断标准
- 主动 grep 是**反"假设式建议"** 的硬性动作

### 决策 6：实现优先级

```
- 阻塞性问题（崩溃、安全）—— 优先
- 简单修复（typo、import）—— 中
- 复杂修复（重构、逻辑）—— 后
```

**设计意图**：
- 阻塞问题**影响最大**——必须先修
- 简单修复**风险最低**——快速清理
- 复杂修复**风险最高**——放到最后，单独评估

**这是 TDD 红绿的"小步走"哲学**——但应用到代码审查。

### 决策 7：反驳的艺术

**何时反驳：**
- 破坏现有功能
- 审查者缺乏完整上下文
- 违反 YAGNI
- 技术上不正确
- 遗留/兼容性原因
- 架构决策冲突

**如何反驳：**
- 用技术推理，而非防御性
- 提出具体问题
- 引用工作中的测试/代码
- 架构层面的让协作伙伴参与

**关键洞察**：
- **"何时反驳"** 提供了**合法性**——不是"教条主义"
- **"如何反驳"** 提供了**方法**——不靠情绪
- **"何时不要反驳"**（暗含）——避免变成"杠精"

### 决策 8：隐语作为"不舒适信号"

```
如果不舒适明说出来的信号：
"Strange things are afoot at the Circle K"（Circle K 发生怪事了）
```

**精妙之处**：
- 用**隐语**告诉伙伴"我有疑虑"但不明说
- 避免直接顶撞审查者（保护关系）
- 让伙伴**主动询问**（避免 AI 主动冲突）

**这是"非对抗性沟通"的高阶技巧**。

### 决策 9：优雅修正反驳

```
✅ "You were right - I checked [X] and it does [Y]. Implementing now."
✅ "Verified this and you're correct. My initial understanding was wrong because [reason]. Fixing."

❌ 长篇道歉
❌ 辩护为什么反驳
❌ 过度解释
```

**关键洞察**：
- 反驳错了**很正常**——不必道歉
- 承认 + 修正 + 继续——**事实性陈述**
- 避免"过度解释"——会让小问题变成大冲突

**这是"建设性纠错"的范例**。

### 决策 10：GitHub 内联回复的具体协议

```
回复到 gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies
不要作为顶级 PR 评论
```

**为什么单独列出**：
- 很多 AI 不知道 GitHub 有专门的"内联回复" API
- 顶级 PR 评论**结构混乱**——所有评论堆一起
- 内联回复**保持线程结构**——便于追踪

**这是"工具特异性知识"的价值**。

## 四、优点

### 1. 精准定位"反 LLM 讨好"行为

**核心问题**：RLHF 训练让 LLM 倾向于"讨好"——这是**模型行为层的根本缺陷**。

**本技能的价值**：把"反讨好"从**软文化**变成**硬规则**。

**与 verification-before-completion 的呼应**：
- verification-before-completion：禁止"虚假完成"
- receiving-code-review：禁止"虚假同意"

**两者共同构成**"AI 行为守门人"的两翼。

### 2. 来源分级是"信任 + 怀疑"的平衡

不是"全信"也不是"全疑"——而是**分级**：
- 协作伙伴：高信任
- 外部审查者：中信任

**这是成熟的人际沟通模式**——应用到 AI 行为。

### 3. 5 步外部验证流程

```text
1. 对本代码库正确吗？
2. 破坏现有功能吗？
3. 当前实现有原因吗？
4. 在所有平台/版本上工作吗？
5. 审查者理解完整上下文吗？
```

**优点**：
- 把"是否实现"从**主观**变成**客观**
- 5 维检查覆盖**技术、组织、平台、上下文**
- 每条都可独立验证

### 4. YAGNI 检查是"反假想需求"

```text
grep 代码库看实际使用情况
```

**关键洞察**：
- 不接受"理论上未来需要"的建议
- 现实使用 = 唯一标准
- 显式 grep = 反"假设式建议"

**这是 YAGNI 哲学的工程化**。

### 5. 反驳合法性 + 反驳方法的双重提供

**何时反驳**（合法性）：
- 破坏现有功能
- 审查者缺乏完整上下文
- ...

**如何反驳**（方法）：
- 用技术推理
- 提出具体问题
- 引用工作中的测试/代码

**优点**：让 AI "**敢于反驳**"——既不教条主义，也不盲从。

### 6. 隐语作为"安全信号"

```
"Strange things are afoot at the Circle K"
```

**优点**：
- 避免直接顶撞审查者
- 给伙伴**主动询问**的机会
- 维护**专业关系**

**这是"软技能"在 AI 行为中的形式化**。

### 7. 优雅修正反驳 = 建设性纠错

```text
✅ "You were right - I checked [X] and it does [Y]. Implementing now."
❌ 长篇道歉
```

**优点**：
- 不浪费**对话篇幅**在情绪上
- 直接进入**问题解决**
- 维护**专业形象**

### 8. GitHub 内联回复的工具特异性

```text
gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies
```

**优点**：
- 不假设"AI 知道所有工具 API"
- 提供**具体命令**
- 让 AI **立刻可用**

## 五、缺点与风险

### 1. "6 步响应模式"可能被 LLM 走形式

**问题**：
- LLM 可以"看起来"走 6 步（READ → UNDERSTAND → ...）
- 但每一步都"快速过场"——实际没认真做
- 形式合规，实质敷衍

**缺失**：**6 步的质量门槛**——什么算"认真"？

### 2. 来源分级过粗

只分 2 类（伙伴/外部），但实际场景复杂：
- 不同伙伴可能不一样（高级 vs 初学者）
- 不同外部审查者不一样（核心维护者 vs 路人）
- 自动化工具（CodeClimate、Copilot）的反馈算什么？

**缺失**：更精细的信任分级。

### 3. "5 步外部验证"可能过度防御

**问题**：
- 每次都跑 5 步——可能拖慢响应
- 对**明显正确**的建议也跑——浪费精力
- 演变成"防御性过度"

**缺失**：**5 步的触发条件**（什么时候跑、什么时候跳过）。

### 4. "何时反驳" 的 6 种情况可能不完整

**未覆盖**：
- 审查者语气不当（人身攻击）
- 审查者偏见
- 审查者使用过时信息
- 审查者利益冲突

**缺失**：**对审查者本身的元判断**——不只是"建议是否对"，还有"审查者是否合适"。

### 5. "禁止感谢"过于绝对

```text
❌ 任何感谢表达
```

**问题**：
- 完全禁止感谢可能显得**冷漠**
- 真实协作中，"Thanks" 是礼貌润滑剂
- 严格执行会让 AI 变成"机器人"

**改进**：区分"表演性感谢"vs"礼貌感谢"——保留后者。

### 6. 缺少"对话管理"建议

**场景**：
- 审查者坚持错误观点
- 协作伙伴和审查者冲突
- 审查者不耐烦

**缺失**：**升级路径**——什么时候引入第三方？什么时候停止讨论？

### 7. "隐语"的文化局限性

```
"Strange things are afoot at the Circle K"
```

**问题**：
- 这是**美国 80 年代 Bill & Ted 电影的台词**
- 非英语文化可能不懂
- 90 后/00 后可能没看过

**改进**：提供**多套隐语**或更通用的"我有疑虑"短语。

### 8. 缺少"反馈质量评估"

**问题**：
- 好的反馈：基于事实、有上下文、有建议
- 差的反馈：基于个人偏好、无依据、含糊

**缺失**：**反馈质量的客观评估**——不只是"是否反驳"。

### 9. 缺少"反馈冲突"的多方处理

**场景**：
- 协作伙伴说"A"
- 外部审查者说"非 A"
- 怎么协调？

**文档只说**："如果与你协作伙伴的先前决策冲突：停下来先和你的协作伙伴讨论"——**过于简化**。

**缺失**：多方冲突的**系统化处理**。

### 10. "GitHub 内联回复"太具体

```text
gh api repos/{owner}/{repo}/pulls/{pr}/comments/{id}/replies
```

**问题**：
- 只适用 GitHub
- GitLab、Bitbucket、Gerrit 类似但命令不同
- 编码 SaaS（Cursor 的 PR）可能完全不同

**缺失**：跨平台 PR 系统的回复协议。

## 六、关键洞察

### 洞察 1：这是"反 LLM 讨好"的核心规范

**观察**：
- RLHF 让 LLM 变得"友好"——但这种"友好"在技术领域是**反生产力**
- "讨好"导致：盲目同意、虚假感谢、未验证的修改
- **训练层面**很难根除——只能**行为层面**防御

**本技能的价值**：把"反讨好"从**软文化**变成**硬规则**。

**与 verification-before-completion 的协同**：
- verification-before-completion：禁止"虚假完成"
- receiving-code-review：禁止"虚假同意"

**两者是 LLM 行为守门人的"双臂"**。

### 洞察 2：6 步响应模式是"决策的形式化"

```text
READ → UNDERSTAND → VERIFY → EVALUATE → RESPOND → IMPLEMENT
```

**关键洞察**：
- 顺序性 = 反"讨好"的防御
- 6 个**显式检查点** = 每一步都是**可观察的**
- 任何步骤不能跳过 = 形式化保证

**这是"AI 决策可审计化"** 的范例。

### 洞察 3：来源分级是"信任的工程化"

**观察**：真实的人际沟通中，信任是**分级**的——不是非黑即白。

**本技能的工程化**：
- 协作伙伴：高信任
- 外部审查者：中信任 + 5 步验证

**关键洞察**：把"信任"从**感觉**变成**协议**。

### 洞察 4：YAGNI 检查是"反假想需求"

```text
grep 代码库看实际使用情况
```

**关键洞察**：
- 现实使用 = 唯一标准
- 显式 grep = 不可绕过
- 这是 **"用户驱动开发"** 在代码审查中的应用

**vs 假想驱动**："未来可能需要"是**最常见**的过度工程借口。

### 洞察 5：反驳的"合法性 + 方法"双重提供

**关键洞察**：
- 只教"如何反驳" → 变成"杠精"
- 只教"何时反驳" → 变成"教条主义"
- 两者都有 → **平衡的批判性思维**

**这是"批判性思维的形式化"**。

### 洞察 6：隐语作为"安全信号"是软技能形式化

```text
"Strange things are afoot at the Circle K"
```

**精妙之处**：
- 避免直接顶撞
- 给伙伴主动询问机会
- 维护专业关系

**本质**：**把"软技能"变成"可学习的行为模式"**。

### 洞察 7："禁止感谢"是反 LLM 讨好文化

```text
❌ 任何感谢表达
```

**精妙之处**：
- "Thanks" 看起来无害——但会**强化讨好模式**
- LLM 学会"Thanks" = LLM 学会"讨好"
- 显式禁止 = 切断强化

**这是"反 LLM 训练偏差"的硬性动作**。

### 洞察 8：与 verification-before-completion 的"二象性"

```
verification-before-completion 关注：结果
receiving-code-review 关注：过程
```

**两者形成"AI 行为守门人"的完整体系**：
- 输出端：禁止虚假完成
- 输入端：禁止虚假同意

**意义**：任何 LLM 应用都应该**同时实施这两条规范**。
