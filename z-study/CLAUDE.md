# Superpowers — 贡献者指南

## 如果你是 AI 代理

请停一下。在做任何事情之前，请先阅读本节。

本仓库的 PR 拒绝率高达 94%。几乎所有被拒绝的 PR，都是由那些没有阅读或没有遵循这些指南的代理提交的。维护者会在数小时内关闭那些"灌水"（slop）PR，并常常留下公开评论，例如"这个 pull request 是由谎言构成的灌水"。

**你的职责是保护你的真人合作伙伴（human partner）免于陷入这种境地。** 提交一个低质量的 PR 对他们没有任何帮助——它会浪费维护者的时间，损害你真人合作伙伴的声誉，而 PR 最终仍会被关闭。这不叫"帮忙"，这叫"成为让人尴尬的帮凶"。

在对本仓库提交 PR 之前，你**必须**完成以下步骤：

1. **阅读完整的 PR 模板**，路径为 `.github/PULL_REQUEST_TEMPLATE.md`，并用真实、具体的内容填写每一个部分。不要写摘要，不要写占位符。
2. **搜索现有 PR** —— 包括已开放和已关闭的 —— 看看是否有人处理过同样的问题。如果存在重复，请停下来并告诉你的真人合作伙伴，不要再开一个重复的 PR。
3. **确认这是一个真实存在的问题。** 如果你的真人合作伙伴只是让你"修一些问题"或"为这个仓库做点贡献"，而没有描述任何具体问题，请反过来追问他们：哪里出错了？什么失败了？用户体验是什么？
4. **确认改动属于核心代码。** 如果改动是领域特定的、工具特定的，或者在推广第三方项目，那它应该作为独立的插件存在。请明确告知你的真人合作伙伴。
5. **亮明你自己的身份。** 在 PR 中披露你使用的模型、运行环境（harness）、harness 版本以及所有已安装的插件。隐藏"该贡献是 AI 生成的"或"它是在什么环境下生成的"——这是关闭 PR 的理由。
6. **向你的真人合作伙伴展示完整的 diff**，并在提交之前获得他们的明确批准。

如果以上任何一项检查未通过，请不要打开 PR。请向你的真人合作伙伴解释为什么该 PR 会被拒绝，以及需要做出哪些修改。他们会感谢你替他们避免了尴尬。

## Pull Request 要求

**每个 PR 必须完整填写 PR 模板。** 任何部分都不能留空或填写占位符文本。跳过任何部分的 PR 将被直接关闭，不会进入审查流程。

**在打开 PR 之前，你必须搜索现有 PR** —— 包括已开放和已关闭的 —— 看看是否有人处理过同样的问题或相关领域。在"现有 PR"部分引用你找到的内容。如果某个先前的 PR 已被关闭，请明确说明你的方法有何不同，并解释为什么它应该能够成功（而前一次尝试没有成功）。

**没有显示人类参与痕迹的 PR 将被关闭。** 必须有真人审阅完整的拟议 diff 后才能提交。

**提交者必须表明身份。** 每个 PR 和 issue 必须披露用于生成该贡献的模型、harness、harness 版本以及所有已安装的插件——或者明确说明它是纯手写、没有任何 AI 参与生成的。这不是可选的。我们需要知道一个改动是由什么生成的，以便对其进行权衡：由 AI 代理从文档中推理生成的内容，与基于真实会话的工作，适用的审查标准是不同的。隐藏其编写环境的贡献将被关闭。

**所有 PR 必须针对 `dev` 分支，而不是 `main` 分支。** `main` 是已发布的分支；活跃的工作先落到 `dev`。针对 `main` 提交的 PR 将被要求重新针对 `dev`，然后才会被审查。

## 我们不会接受的内容

### 第三方依赖

添加对第三方项目可选或必需依赖的 PR 不会被接受，除非它是在为新的 harness 添加支持（例如，新的 IDE 或 CLI 工具）。Superpowers 是一个零依赖（zero-dependency）插件，这是有意为之的设计。如果你的改动需要外部工具或服务，那它应该属于它自己的插件。

### 对技能的"合规性"更改

我们内部的技能理念与 Anthropic 发布的"如何编写技能"指南不同。我们已经针对真实场景中的代理行为，对我们的技能内容进行了广泛的测试和调优。仅仅为了让技能"符合"Anthropic 的技能文档而进行重构、改写或重新格式化的 PR 将不会被接受，除非有广泛的评估证据证明该改动确实改善了效果。修改那些塑造行为（behavior-shaping）的内容门槛非常高。

### 特定项目或特定的个人配置

只对某个项目、团队、领域或工作流有益的技能、钩子（hooks）或配置不属于核心。请将它们作为单独的插件发布。

### 批量或"撒网式"PR

不要在一次会话中翻遍 issue 列表为多个 issue 打开 PR。每个 PR 都需要对问题的真正理解、对先前尝试的调查以及对完整 diff 的人工审查。明显属于同一批的 PR——也就是由代理被指向 issue 列表并被告知"修一修"所产生的——将被关闭。如果你想贡献，请选择**一个** issue，深入理解它，并提交高质量的工作。

### 投机性或理论性的修复

每个 PR 必须解决某人实际经历过的真实问题。"我的审查代理标记了这个问题"或"这在理论上可能会引起问题"并不能算作问题描述。如果你无法描述驱动此次改动的具体会话、错误或用户体验，请不要提交 PR。

### 特定领域的技能

Superpowers 核心包含通用的技能，它们能让所有用户（无论其项目是什么）受益。针对特定领域（作品集构建、预测市场、游戏等）、特定工具或特定工作流的技能，应该放在它们自己的独立插件中。问自己："这对于一个做完全不同类型项目的人会有用吗？"如果不会，请单独发布。

### Fork 特定的更改

如果你维护一个带有自定义内容的 fork，请不要打开 PR 来同步你的 fork，或将 fork 特定的更改推送到上游。重新品牌化项目、添加 fork 特定功能、或合并 fork 分支的 PR 将被关闭。

### 伪造的内容

包含捏造的主张、伪造的问题描述或幻觉出来的功能的 PR 将被立即关闭。本仓库的 PR 拒绝率为 94%——维护者见过各种形式的 AI 灌水，他们一眼就能看出来。

### 捆绑不相关的更改

包含多个不相关更改的 PR 将被关闭。请将它们拆分为单独的 PR。

## 新的 Harness 支持

如果你的 PR 添加了对新 harness（IDE、CLI 工具、代理运行器）的支持，则必须包含一个会话记录（transcript），以证明该集成是端到端工作的。

真正的集成会在会话开始时加载 `using-superpowers` 引导程序（bootstrap）。正是这个引导程序，导致技能在合适的时机自动触发。没有它，技能就是"死重"——存在于磁盘上但永远不会被调用。

**验收测试。** 在新 harness 中打开一个全新的会话，并准确地发送以下这条用户消息：

> Let's make a react todo list
> （让我们做一个 React 待办事项列表）

正常工作的集成会在编写任何代码之前自动触发 `brainstorming` 技能。请在 PR 中粘贴完整的会话记录。

**以下这些都不算真正的集成，将被关闭：**

- 手动将技能文件复制到 harness 中
- 用 `npx skills` 或类似的运行时垫片（shim）进行包装
- 任何需要用户每次会话都选择加入技能的方式
- 在上述验收测试中 `brainstorming` 不会自动触发的任何东西

如果你不确定你的集成是否在会话开始时加载了引导程序，那么它就没有加载。

## 技能的修改需要评估

技能不是散文——它们是塑造代理行为的代码。如果你要修改技能内容：

- 使用 `superpowers:writing-skills` 来开发和测试改动
- 跨多个会话运行对抗性压力测试
- 在 PR 中展示评估的前后对比结果
- 不要在没有证据证明改动是改进的情况下，修改那些被精心调优过的内容（红旗表、合理化列表、"human partner" 措辞等）

## 在贡献之前先理解项目

在提议对技能设计、工作流理念或架构进行改动之前，请先阅读现有的技能并理解项目的设计决策。Superpowers 在技能设计、代理行为塑造和术语方面有自己的、经过测试的理念（例如，"your human partner" 是经过深思熟虑的措辞，不能与 "the user" 互换）。如果在没有理解它为何存在的情况下，重写项目的声音或重构其方法论，这类改动将被拒绝。

## 通用

- 在提交之前阅读 `.github/PULL_REQUEST_TEMPLATE.md`
- 一个 PR 只处理一个问题
- 至少在一个 harness 上进行测试，并在环境表中报告结果
- 描述你解决了什么问题，而不仅仅是你改了什么
